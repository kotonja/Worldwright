# Playtest Critic 0.1.0

## Status and scope

`@worldwright/playtest-critic` `0.1.0` defines Worldwright's first source-bound architectural
Playtest Plan, sanitized Playtest Run Report, and deterministic Critic Report. It evaluates live
reachability and integrity evidence without depending on Studio or the MCP SDK and without
generating a repair.

This reference is normative for the `0.1.0` wire meanings. TypeBox source files are canonical; JSON
Schemas are generated artifacts.

## Contract identifiers

| Contract            | Version | JSON Schema ID                              |
| ------------------- | ------- | ------------------------------------------- |
| Playtest Plan       | `0.1.0` | `urn:worldwright:playtest-plan:0.1.0`       |
| Playtest Run Report | `0.1.0` | `urn:worldwright:playtest-run-report:0.1.0` |
| Critic Report       | `0.1.0` | `urn:worldwright:critic-report:0.1.0`       |

Every domain object is closed with `additionalProperties: false`. Public boundaries reject
non-finite numbers, `undefined`, BigInt, functions, symbols, sparse arrays, accessors,
non-enumerable custom properties, cyclic data, class instances, executable source, credentials, raw
MCP payloads, raw Studio logs, local paths, image bytes, Studio IDs, lease IDs, and environment
data.

## Planning inputs and source record

Planning accepts one valid Architecture Plan `0.1.0` and the exact validated Roblox Manifest `0.1.0`
to be exercised. It does not mutate either input. The Playtest Plan source records:

- Architecture Plan schema and planner versions;
- canonical Architecture Plan SHA-256;
- authored source WorldSpec SHA-256;
- project ID and building entity ID;
- Manifest schema and compiler versions;
- canonical Manifest SHA-256;
- Manifest source WorldSpec SHA-256;
- Manifest root node ID; and
- expected managed instance count.

Planning requires matching project identity, exact root source metadata, and complete correspondence
for the plan-emitted building, room containers, floors, walls, openings, corridors, stair halls,
landings, and stair-run geometry. Stable IDs and exact managed properties are evidence; display
names and visual similarity are not.

The authored source hash in the Architecture Plan and derived source hash in the Manifest identify
different WorldSpecs. The current closed Manifest does not carry the plan hash or authored source
hash, so version `0.1.0` cannot claim cryptographic Plan-to-Manifest derivation. It binds both
canonical artifact hashes and fails closed on exact structural correspondence.

## Playtest Plan

The top-level Plan contains `schemaVersion`, `criticVersion`, `source`, `agent`, `setup`,
`checkpoints`, `segments`, `captureCheckpoints`, `requiredCoverage`, and `limits`.

### Agent profile

Version `0.1.0` has one fixed solo-character profile:

| Field                                 | Value             |
| ------------------------------------- | ----------------- |
| Radius                                | `2` studs         |
| Height                                | `5` studs         |
| Can jump                              | `false`           |
| Can climb                             | `false`           |
| Waypoint spacing                      | `4` studs         |
| Horizontal arrival tolerance          | `4` studs         |
| Vertical arrival tolerance            | `5` studs         |
| Maximum accepted horizontal speed     | `32` studs/second |
| Maximum fall below the floor envelope | `12` studs        |
| Root height above finished floor      | `3` studs         |

These values are a deterministic test profile, not an accessibility or building-code claim.

### Setup

The Plan identifies one exterior setup checkpoint and position, expected floor and level, entrance
opening, entrance room, and finished-floor elevation. Setup is always marked excluded from scoring.

### Checkpoint types

| Type                | Meaning                                                                   |
| ------------------- | ------------------------------------------------------------------------- |
| `exterior_entrance` | Safe exterior side of the explicit entrance opening.                      |
| `opening_threshold` | Safe side-specific point adjacent to an explicit door or open connection. |
| `room_center`       | Center of one source room clear-space rectangle.                          |
| `corridor`          | Safe corridor point associated with explicit circulation.                 |
| `stair_hall`        | Safe point in a participating floor's stair hall.                         |
| `stair_landing`     | Clear retained lower or upper landing center for one stair run.           |

Every checkpoint binds deterministic ID, type, semantic and floor IDs, level, local and world
position, expected floor elevation, required status, and the exact applicable room, opening,
corridor, circulation-node, or stair-run IDs.

### Segments and route

Every segment has deterministic ID and sequence, from/to checkpoint IDs, source circulation edge,
traversal type `door`, `open`, `corridor`, or `stair`, expected levels, one maximum navigation
attempt, and required path, arrival, and clearance checks.

Targets are ordered exterior entrance, entrance room, ascending floor level, zone order public then
service then private, and source room ID by Unicode code point. Iterative breadth-first search sorts
neighbors by code-point ID and selects the deterministic shortest route. It removes only consecutive
duplicate checkpoints and preserves required repeated corridor or stair traversal.

### Required coverage and limits

Coverage records exact sorted IDs and counts for required rooms, floors, corridors, stair runs,
openings, checkpoints, and segments. Counts are recomputed and must match their ID sets and route.

| Limit                                   | Value        |
| --------------------------------------- | ------------ |
| Checkpoints                             | `128`        |
| Route segments                          | `256`        |
| Captures                                | `8`          |
| Retained path waypoints per segment     | `128`        |
| Raw console entries observed            | `512`        |
| Sanitized console entries in the report | `64`         |
| Navigation wait per segment             | `45,000 ms`  |
| Total playtest wait                     | `900,000 ms` |
| Character-load wait                     | `60,000 ms`  |
| Start or Stop state-transition wait     | `60,000 ms`  |

User input cannot expand these bounds in version `0.1.0`.

## Coordinate semantics

Local X/Z coordinates use the Architecture Plan footprint-centered frame. World Y uses exact
finished-floor elevation plus the root-height offset. Horizontal local-to-world mapping uses the
planner's exact quarter-turn formulas for yaw `0`, `90`, `180`, and `270`. Negative zero is
normalized.

Room checkpoints use clear-rectangle centers. Door thresholds use canonical wall axis, opening
center, wall thickness, and a fixed face offset. Corridor points remain inside corridor clear space.
Stair landings use the retained clear lower and upper landing centers. Semantic validation rejects
any point inside wall, glass, slab, step, blocked edge, outside its clear rectangle, or in a
wrong-floor vertical zone.

See [the traversal model](traversal-model.md) for graph construction and coverage rules.

## Playtest Run Report

The Run Report records:

- exact plan, architecture, Manifest, project, root, and source hashes;
- zero PlaceId/GameId facts, pre-play Edit hash, managed count, data models used, and boolean exact
  Studio/lease verification facts without private IDs;
- start acknowledgment and observed-state evidence;
- verified, score-excluded setup evidence;
- one normalized result for every planned segment;
- bounded sanitized console and viewport evidence;
- recomputed checkpoint, room, floor, and stair coverage;
- Stop acknowledgment and observed Edit restoration; and
- exact pre/post Edit hashes and final Manifest no-op count.

Per-segment evidence separates path result, navigation request, independent arrival, character
state, and clearance. Tagged `not_attempted` or `not_observed` variants preserve truthful aborted
runs. Run summary status is `completed`, `aborted`, `failed_to_start`, or `failed_to_stop` and
contains no timestamp or wall-clock duration.

See [the evidence model](evidence-model.md) for privacy and completeness rules.

## Critic Report

The Critic Report binds the canonical Plan and Run Report hashes and contains `pass`,
`pass_with_warnings`, or `fail`, sorted findings, exact metrics, and evidence-completeness facts.

Every finding has deterministic ID, closed code, severity, category, fixed concise message, optional
related floor, sorted related source/checkpoint/segment/evidence IDs, and one closed suggestion
code. Messages and suggestions are fixed rule metadata, not caller prose.

### Hard findings

Hard rules cover at least:

- incomplete run or unproved play-simulation identity;
- start, character load, or setup failure;
- character death, zero health, or excessive fall;
- unsuccessful required path or a jump waypoint under the non-jumping agent;
- arrival outside tolerance or wrong floor after stairs;
- missed required checkpoint, room, floor, or stair run;
- missing support or blocked head/body clearance;
- new console error or incomplete console evidence;
- unsafe Stop, failure to restore Edit, changed Edit snapshot hash; and
- nonzero final Manifest reconciliation.

The closed hard finding codes are:

- `critic.run_incomplete`
- `critic.play_simulation_identity_unproved`
- `critic.playtest_start_failed`
- `critic.character_missing`
- `critic.character_dead`
- `critic.character_fell`
- `critic.setup_failed`
- `critic.path_not_successful`
- `critic.path_requires_jump`
- `critic.arrival_not_reached`
- `critic.wrong_floor`
- `critic.checkpoint_not_reached`
- `critic.room_not_reached`
- `critic.floor_not_reached`
- `critic.stair_not_traversed`
- `critic.head_clearance_blocked`
- `critic.body_clearance_blocked`
- `critic.support_missing`
- `critic.console_error_new`
- `critic.console_evidence_incomplete`
- `critic.playtest_stop_failed`
- `critic.edit_not_restored`
- `critic.edit_snapshot_changed`
- `critic.manifest_not_noop`

### Warning findings

Warnings cover at least:

- a new console warning;
- arrival within tolerance with excessive final velocity;
- a nearby unmanaged blocker that did not prevent arrival;
- uncertain navigation acknowledgment with independently proved arrival;
- disproportionate path detour; and
- unavailable evidence at a nonessential capture checkpoint.

The closed warning finding codes are:

- `critic.console_warning_new`
- `critic.arrival_velocity_high`
- `critic.unmanaged_blocker_nearby`
- `critic.navigation_ack_uncertain`
- `critic.path_detour_high`
- `critic.capture_unavailable`

### Suggestion codes

The closed descriptive suggestions are:

- `inspect-opening-clearance`
- `widen-corridor`
- `inspect-stair-geometry`
- `inspect-floor-support`
- `inspect-collision-volume`
- `inspect-spawn-or-setup`
- `inspect-console-error`
- `rerun-with-complete-evidence`
- `restore-edit-snapshot`
- `inspect-playtest-state`

Suggestions are review hints. They are not executable instructions, repair authorization, or Change
Sets.

### Status and ordering

Any error produces `fail`. If every hard rule passes and at least one warning exists, status is
`pass_with_warnings`. `pass` requires a completed, fully covered, surviving, path/arrival/clearance-
clean run; complete console evidence with no new error or warning; verified Edit restoration; exact
snapshot equality; and zero final reconciliation operations.

Findings sort by error before warning, category, related floor, related source ID, code, then
finding ID using Unicode code-point comparison.

## Normalization, validation, and hashing

Schema validation precedes semantic validation. Semantic checks resolve every reference, reject
duplicates, require contiguous route and result sequences, recompute all counts and metrics, and
compare run source and coverage against the Plan. A Critic artifact can also be re-evaluated against
its Plan and Run Report and compared canonically.

Normalization creates a deep-independent value. Set-like IDs sort by Unicode code point; route and
segment results preserve sequence; findings use Critic order. Canonical JSON recursively sorts
object keys by code point, uses two spaces, and ends with one line feed. Hashes are lowercase
SHA-256 over that normalized representation.

## CLI

Offline commands are:

```sh
pnpm playtest-critic plan <architecture-plan-file> --manifest <manifest-file>
pnpm playtest-critic plan <architecture-plan-file> --manifest <manifest-file> --output <playtest-plan-file>
pnpm playtest-critic evaluate --plan <playtest-plan-file> --run-report <run-report-file>
pnpm playtest-critic evaluate --plan <playtest-plan-file> --run-report <run-report-file> --output <critic-report-file>
```

`--json` emits canonical machine-readable success or diagnostics where supported. Artifact output
paths must differ from input paths.

The privileged live workflow is a separate Studio command so the pure package does not depend on the
Studio adapter:

```sh
pnpm studio:playtest-live-smoke -- --review
pnpm studio:playtest-live-smoke -- \
  --studio-id <exact-private-id> \
  --sandbox-lease-id <private-64-lowercase-hex> \
  --confirm <full-reviewed-sequence-sha256> \
  --confirm-plan <exact-playtest-plan-sha256> \
  --confirm-change-set <complete-change-set-sha256>
```

`--review` prints those three hashes without opening MCP and without emitting a Studio ID,
sandbox-lease identity, or local path.

| Exit | Meaning                                                                     |
| ---- | --------------------------------------------------------------------------- |
| `0`  | Planning succeeded, or evaluation/live run contains no hard Critic finding. |
| `1`  | Domain input, source binding, run, or Critic failed.                        |
| `2`  | Usage, I/O, MCP startup, capability, or exact-session selection failed.     |

## Security and performance

The pure package performs no network or Studio operation. It accepts no scripts, credentials,
arbitrary code, image bytes, raw logs, or private Studio identity. Planning and evaluation are
bounded, iterative, deterministic, and non-mutating. They precompute checkpoint, graph, floor, room,
opening, stair, capture, and route maps. They do not use random values, system time,
locale-sensitive comparison, unbounded polling, recursive traversal dependent on project size, or
all-pairs route recomputation.

## Limitations and versioning

Version `0.1.0` supports the Architecture Planner's bounded double-loaded-spine blockout and one
solo test character. It does not prove accessibility, code compliance, beauty, style, image
fidelity, lighting, furnishing, terrain, gameplay fun, multiplayer behavior, performance, or publish
readiness. Captures have no visual interpretation.

Patch releases may fix implementation or prose without changing accepted shape or meaning.
Checkpoint coordinates, route ordering, agent values, evidence semantics, finding rules, messages,
suggestions, status rules, ordering, and canonical hash inputs are observable behavior and require
an intentional version decision when changed. Existing WorldSpec, Architecture Plan, Manifest,
Snapshot, Change Set, Studio bridge, receipt, batch, lease, progress, and transport contracts retain
their meanings.
