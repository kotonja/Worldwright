# WorldSpec 0.1.0

WorldSpec is Worldwright's versioned, JSON-serializable semantic representation of a world. It is
the contract that allows reasoning, planning, compilation, creator tooling, evaluation, and repair
systems to exchange intent without depending on one language, process, AI provider, or Roblox
runtime representation.

This document describes WorldSpec version `0.1.0`:

- **Schema version:** `0.1.0`
- **Schema ID:** `urn:worldwright:worldspec:0.1.0`
- **JSON Schema dialect:** draft 2020-12
- **Package:** `@worldwright/worldspec`

WorldSpec v0.1 is a semantic plan. It can describe that a staircase connects floors or that a facade
must be preserved; it does not contain the mesh, Instances, scripts, or compiler operations that
implement those ideas.

## JSON data contract

A WorldSpec document must be ordinary JSON data. The in-memory validator rejects values that cannot
be represented faithfully in JSON, including:

- functions and symbols;
- class instances and `Date` objects;
- `BigInt`;
- `undefined` values;
- `NaN`; and
- positive or negative `Infinity`.

All defined domain objects are closed: unknown properties are rejected. The only intentionally
open-ended objects are explicitly named JSON-value maps, currently entity and relationship
`attributes` and constraint `parameters`. Values inside those maps must still be JSON-compatible.

The exact-integer fields `project.seed`, `budgets.limits.instances`, and `budgets.limits.triangles`
have a maximum of `9,007,199,254,740,991`, the largest integer JSON consumers based on IEEE 754
binary64 numbers can represent exactly. This prevents a seed or count from being silently rounded as
it crosses a JavaScript or JSON-tool boundary. Extension data that needs a larger exact integer
should encode it as a string until a typed, versioned field defines its representation. This maximum
does not restrict floating-point fields such as transforms, `influence`, `confidence`, or
`textureMemoryMegabytes`; those retain their field-specific ranges.

## Top-level document

Every field in the top-level object is required.

| Field           | Type              | Purpose                                                                                    |
| --------------- | ----------------- | ------------------------------------------------------------------------------------------ |
| `schemaVersion` | literal `"0.1.0"` | Selects the exact contract version. Unsupported versions are not coerced.                  |
| `project`       | `Project`         | Stable project identity and coordinate conventions.                                        |
| `intent`        | `Intent`          | Human-authored summary, requirements, exclusions, and preferences.                         |
| `references`    | `Reference[]`     | Ordered source material used to understand or influence the world.                         |
| `style`         | `StyleDna`        | Ordered architectural, shape, material, palette, lighting, and exclusion guidance.         |
| `rootEntityId`  | identifier        | Identifies the single root world entity.                                                   |
| `entities`      | `Entity[]`        | Semantic hierarchy of world, region, structure, space, route, and object concepts.         |
| `relationships` | `Relationship[]`  | Cross-hierarchy semantic connections between entities.                                     |
| `constraints`   | `Constraint[]`    | Error- or warning-level requirements that planning, compilation, or critique can evaluate. |
| `locks`         | `Lock[]`          | Creator or system protections on selected entity fields.                                   |
| `budgets`       | `Budgets`         | Target-device, quality, streaming, and optional content limits.                            |

## Core objects

### Project

`project` contains:

| Field         | Type                       | Notes                                                                                          |
| ------------- | -------------------------- | ---------------------------------------------------------------------------------------------- |
| `id`          | identifier                 | Participates in global ID uniqueness.                                                          |
| `name`        | non-empty string           | Human-readable project name.                                                                   |
| `description` | non-empty string, optional | Human-readable context.                                                                        |
| `seed`        | non-negative safe integer  | Stable seed, at most `9,007,199,254,740,991`, for deterministic future planners and compilers. |
| `units`       | literal `"studs"`          | WorldSpec v0.1's spatial unit.                                                                 |
| `upAxis`      | literal `"Y"`              | WorldSpec v0.1's vertical axis.                                                                |

The seed does not make an otherwise nondeterministic external system deterministic; it is a recorded
input that such systems must honor when their milestone defines seed behavior.

### Intent

`intent` contains:

- `summary`: a non-empty overview of the desired world;
- `mustHave`: an ordered list of requirements;
- `mustNotHave`: an ordered list of exclusions; and
- `preferences`: an ordered list of softer preferences.

These arrays preserve author order because priority, presentation, or narrative sequence may be
meaningful. A later solver must not reinterpret a preference as a hard requirement merely because it
appears in this object.

### References

Each reference contains:

| Field       | Type                               | Notes                                                                                   |
| ----------- | ---------------------------------- | --------------------------------------------------------------------------------------- |
| `id`        | identifier                         | Stable reference ID; participates in global uniqueness.                                 |
| `kind`      | reference-kind enum                | `image`, `sketch`, `floor_plan`, `heightmap`, `existing_place`, or `text`.              |
| `role`      | non-empty string                   | Describes why the reference is present, such as exterior massing or library atmosphere. |
| `uri`       | non-empty string, optional         | A locator; WorldSpec does not fetch it. Avoid embedding secrets.                        |
| `influence` | finite number from `0` through `1` | Relative influence declared by the author or an upstream system.                        |
| `notes`     | non-empty string, optional         | Additional context.                                                                     |

The `references` array preserves author order. A URI is data only: validating a document performs no
network access and does not establish that the resource exists or is trusted.

### Style DNA

`style` provides author-directed visual guidance:

- `architecture`: ordered architectural styles or influences;
- `shapeLanguage`: ordered form, silhouette, rhythm, and proportion language;
- `materialFamilies`: ordered material families;
- `palette`: ordered color or palette guidance;
- `detailDensity`: `low`, `medium`, `high`, or `hero`;
- `aging`: `pristine`, `light`, `moderate`, or `heavy`;
- `lighting`: ordered lighting direction; and
- `exclusions`: ordered visual choices to avoid.

Style arrays preserve author order. WorldSpec v0.1 records these descriptors but does not define a
renderer, asset lookup, color-management model, or automated measure of style similarity.

## Entities and hierarchy

An entity is a semantic world element. Its `kind` is one of:

| Kind          | Intended use                                                      |
| ------------- | ----------------------------------------------------------------- |
| `world`       | The single root of the entity hierarchy.                          |
| `region`      | A broad environmental or geographic area.                         |
| `district`    | A subdivision with shared spatial or functional identity.         |
| `parcel`      | A bounded site or lot.                                            |
| `structure`   | A building or other major constructed whole.                      |
| `floor`       | A semantic level within a structure.                              |
| `room`        | A bounded interior or functionally distinct space.                |
| `route`       | A circulation element such as a stair, path, corridor, or bridge. |
| `terrain`     | A semantic terrain feature.                                       |
| `landmark`    | A prominent orientation or composition element.                   |
| `object`      | A semantic object not represented by a more specific kind.        |
| `spawn`       | A player or actor spawn concept.                                  |
| `interaction` | An interactive semantic element or affordance.                    |

Every entity contains:

| Field        | Type                          | Notes                                                                      |
| ------------ | ----------------------------- | -------------------------------------------------------------------------- |
| `id`         | identifier                    | Stable entity ID; participates in global uniqueness.                       |
| `kind`       | entity-kind enum              | One of the kinds above.                                                    |
| `name`       | non-empty string              | Human-readable name.                                                       |
| `parentId`   | entity identifier, optional   | Required semantically for every non-root entity and forbidden on the root. |
| `provenance` | `Provenance`                  | Records evidence classification and confidence.                            |
| `transform`  | `Transform`, optional         | Semantic placement when known.                                             |
| `bounds`     | `Bounds`, optional            | Positive axis-aligned semantic size when known.                            |
| `tags`       | unique non-empty string array | Set-like labels; normalized lexicographically.                             |
| `attributes` | JSON-value map                | Explicit extension data for downstream milestones.                         |

The entity array is not the hierarchy by position. `rootEntityId` selects the root and `parentId`
defines parent edges. Semantic validation requires exactly the useful rooted structure: the root
exists, is a `world`, has no parent, and every other entity has a valid parent and eventually
reaches the root without a cycle.

Entity transforms do not currently define local-versus-world composition semantics beyond recording
the values. A future planning or compiler milestone must establish those rules before depending on
nested transform math.

### Provenance

Every entity's provenance contains:

- `classification`: `observed`, `inferred`, or `invented`;
- `referenceIds`: a set-like list of IDs from the top-level `references` array;
- `confidence`: a finite number from `0` through `1`; and
- optional non-empty `notes`.

The classifications are intentionally distinct:

- **Observed** means the stated feature is directly supported by source evidence.
- **Inferred** means the feature is a reasoned interpretation or completion from incomplete
  evidence.
- **Invented** means the feature was introduced to satisfy intent, coherence, traversal, gameplay,
  or another requirement without direct reference evidence.

A feature hidden behind a facade in an image cannot truthfully be marked observed merely because an
upstream model produced it confidently. `confidence` expresses certainty within the chosen
classification; it does not change the classification.

### Transform and bounds

A transform contains all three vectors:

```json
{
  "position": { "x": 0, "y": 12, "z": -40 },
  "rotationEulerDegrees": { "x": 0, "y": 90, "z": 0 },
  "scale": { "x": 1, "y": 1, "z": 1 }
}
```

Every vector coordinate must be finite. Scale components must be greater than zero.

Bounds contain a positive size:

```json
{
  "size": { "x": 120, "y": 48, "z": 80 }
}
```

Every size component must be finite and greater than zero. Bounds are semantic planning bounds, not
generated mesh data.

## Relationships

Relationships express semantic connections that are not reducible to the parent hierarchy. Each
relationship contains:

- `id`: a globally unique identifier;
- `type`: `contains`, `adjacent_to`, `connects_to`, `supports`, `depends_on`, `visible_from`,
  `serves`, or `aligned_with`;
- `sourceId` and `targetId`: existing entity IDs;
- `directed`: a boolean describing whether source-to-target direction is meaningful; and
- `attributes`: an open JSON-value map for explicit downstream metadata.

A relationship cannot connect an entity to itself. The validator checks endpoints and
self-relationships, but v0.1 does not infer inverse edges, enforce type-specific entity kinds, or
prove that geometry realizes the relationship.

## Constraints

Constraints record requirements for future planning, compilation, testing, or critique. Each
constraint contains:

| Field         | Type                 | Notes                                                                                                                      |
| ------------- | -------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `id`          | identifier           | Globally unique constraint ID.                                                                                             |
| `type`        | constraint-type enum | `reachability`, `adjacency`, `clearance`, `containment`, `alignment`, `preservation`, `performance`, `style`, or `custom`. |
| `severity`    | `error` or `warning` | Intended consequence if the requirement is not met.                                                                        |
| `source`      | source enum          | `user`, `system`, `reference`, or `inference`.                                                                             |
| `description` | non-empty string     | Human-readable requirement.                                                                                                |
| `subjectIds`  | entity ID array      | Set-like entities to which the constraint applies.                                                                         |
| `targetIds`   | entity ID array      | Set-like comparison, destination, or related entities.                                                                     |
| `parameters`  | JSON-value map       | Type-specific values that remain data, never executable code.                                                              |

Semantic validation verifies that subject and target IDs exist. WorldSpec v0.1 does not implement a
solver or define a universal parameter vocabulary for each constraint type. A future solver
milestone must document the parameter contracts it supports and must report unsupported constraints
instead of pretending they were enforced.

## Locks

Locks protect intentional parts of a plan from broad regeneration or repair. Each lock contains:

- `id`: a globally unique identifier;
- `entityId`: an existing entity;
- `fieldPaths`: one or more non-empty field paths;
- `owner`: `user` or `system`; and
- optional non-empty `reason`.

`fieldPaths` are set-like and normalized lexicographically. They identify protected fields for
future tools; WorldSpec v0.1 validates that at least one non-empty path exists but does not apply
locks or define wildcard/path-resolution behavior. A future mutation protocol must define matching
and conflict handling before locks can authorize or reject edits.

## Budgets

`budgets` contains:

| Field           | Type                     | Notes                                                              |
| --------------- | ------------------------ | ------------------------------------------------------------------ |
| `targetDevices` | non-empty device array   | Set-like selection of `desktop`, `mobile`, `console`, and/or `vr`. |
| `qualityTier`   | quality enum             | `draft`, `standard`, `high`, or `cinematic`.                       |
| `streaming`     | streaming enum           | `disabled`, `preferred`, or `required`.                            |
| `limits`        | `BudgetLimits`, optional | Positive content ceilings described below.                         |

When present, `limits` is a closed object with optional positive values:

| Field                    | Type                             | Notes                                                                            |
| ------------------------ | -------------------------------- | -------------------------------------------------------------------------------- |
| `instances`              | positive safe integer, optional  | Planned upper limit on Instance count; maximum `9,007,199,254,740,991`.          |
| `triangles`              | positive safe integer, optional  | Planned upper limit on rendered triangle count; maximum `9,007,199,254,740,991`. |
| `textureMemoryMegabytes` | positive finite number, optional | Planned texture-memory limit in megabytes.                                       |

Budgets are declared requirements, not measurements. Milestone 0 validates their shape; no compiler
or Critic yet enforces or measures them.

## Identifier rules

WorldSpec uses stable, human-readable, prefixed identifiers rather than database-generated numeric
IDs.

An identifier:

- is 1 through 128 characters;
- matches `^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$`;
- begins with a lowercase ASCII letter;
- uses lowercase ASCII letters, digits, and single hyphen separators;
- contains no whitespace, underscores, punctuation, leading or trailing hyphen, or consecutive
  hyphens;
- is compared exactly and case-sensitively; and
- is globally unique across the project, references, entities, relationships, constraints, and
  locks.

Although the schema only accepts lowercase identifiers, consumers must still use exact string
comparison and must not apply case folding. The format is safe to place in JSON keys and logs and to
carry as the value of a future Roblox attribute.

Use a category prefix that remains meaningful in logs and diagnostics:

```text
project-cliffside-mansion
ref-mansion-front
entity-world
entity-primary-mansion
relationship-foyer-ballroom
constraint-all-rooms-reachable
lock-front-facade
```

IDs are identity, not display names. Renaming an entity does not require changing its ID, and array
position never supplies identity.

## Validation model

Validation accepts `unknown` and never mutates the input. Invalid documents return diagnostics
rather than requiring callers to catch validation exceptions or parse Ajv messages.

### Phase 1: schema and JSON-value validation

The schema phase checks:

- JSON compatibility of the in-memory value;
- supported `schemaVersion`;
- required and unknown properties;
- primitive and object shape;
- enum values;
- identifier format;
- string and array cardinality;
- finite numeric ranges; and
- positive scale, size, and budget limits.

If the schema phase fails, semantic validation is not run on a value whose structure cannot be
trusted. Schema failures use `schema.invalid`; invalid JSON text parsed through `parseWorldSpec`
uses `json.invalid`.

### Phase 2: semantic validation

For a schema-valid document, the semantic phase checks:

1. IDs are globally unique across project, references, entities, relationships, constraints, and
   locks.
2. `rootEntityId` points to an existing entity.
3. The root entity has kind `world`.
4. The root entity has no parent.
5. Every non-root entity has a parent.
6. Every `parentId` points to an existing entity.
7. An entity cannot parent itself.
8. The parent graph contains no cycles.
9. Every entity is ultimately reachable from the selected root through its parent chain.
10. Every provenance `referenceId` points to an existing reference.
11. Every relationship endpoint points to an existing entity.
12. A relationship cannot connect an entity to itself.
13. Every constraint subject and target points to an existing entity.
14. Every lock targets an existing entity.
15. Every lock has at least one non-empty field path.

Finite-number requirements are enforced during the first phase, including for in-memory values that
JSON text itself could not represent.

### Diagnostics

A diagnostic is a stable object:

```ts
interface Diagnostic {
  readonly code: string;
  readonly severity: 'error' | 'warning';
  readonly path: string;
  readonly message: string;
  readonly relatedId?: string;
}
```

- `code` is the machine-readable value automation should branch on.
- `severity` describes impact. Invalid v0.1 documents currently produce error diagnostics; the shape
  allows future non-fatal findings.
- `path` is a JSON Pointer-like location. The document root is the empty string `""`; escaped path
  segments use `~0` for `~` and `~1` for `/`. It identifies the nearest useful document location
  rather than exposing a library-specific path. Human formatting displays the empty root as `/`.
- `message` is concise text for people and may improve without changing the code.
- `relatedId`, when present, ties the finding to the relevant semantic ID.

Do not parse `message` in automation. Diagnostic ordering is deterministic.

The v0.1 diagnostic code catalog is:

| Code                            | Meaning                                                       |
| ------------------------------- | ------------------------------------------------------------- |
| `json.invalid`                  | Input text is not valid JSON.                                 |
| `schema.invalid`                | The value violates the v0.1 schema or JSON-value contract.    |
| `id.duplicate`                  | An ID is reused anywhere in the global ID namespace.          |
| `entity.root_missing`           | `rootEntityId` does not select an existing entity.            |
| `entity.root_wrong_kind`        | The selected root is not a `world`.                           |
| `entity.root_has_parent`        | The selected root declares a parent.                          |
| `entity.parent_missing`         | A non-root has no parent or a declared parent does not exist. |
| `entity.parent_self`            | An entity names itself as parent.                             |
| `entity.parent_cycle`           | The entity parent graph contains a cycle.                     |
| `entity.unreachable`            | An entity does not ultimately reach the selected root.        |
| `reference.missing`             | Entity provenance names an unknown reference.                 |
| `relationship.endpoint_missing` | A relationship source or target is unknown.                   |
| `relationship.self`             | A relationship connects an entity to itself.                  |
| `constraint.entity_missing`     | A constraint subject or target is unknown.                    |
| `lock.entity_missing`           | A lock targets an unknown entity.                             |
| `lock.path_empty`               | A lock contains no usable field path.                         |

The schema code intentionally summarizes third-party validation keywords behind a stable Worldwright
boundary. Individual schema diagnostics still carry a useful path and message.

## Public package API

The package exports the following intentional surface:

- `WORLD_SPEC_VERSION` — the literal supported version, `0.1.0`.
- `WORLD_SPEC_SCHEMA_ID` — `urn:worldwright:worldspec:0.1.0`.
- `WorldSpecSchema` — the TypeBox runtime source schema.
- Schema-derived static types: `WorldSpec`, `Project`, `Intent`, `WorldReference`, `StyleDna`,
  `Provenance`, `Transform`, `Bounds`, `WorldEntity`, `WorldRelationship`, `WorldConstraint`,
  `WorldLock`, and `WorldBudgets`, plus the recursive `JsonValue` type.
- `validateWorldSpec(input: unknown)` — validate an in-memory value and return a `ValidationResult`.
- `parseWorldSpec(source: unknown)` — require and parse a JSON string, validate it, and return a
  `ValidationResult` without throwing for a non-string source, invalid JSON, or an invalid document.
- `normalizeWorldSpec(worldSpec: Readonly<WorldSpec>)` — return a deep, independent,
  deterministically normalized value without mutating the input.
- `stringifyWorldSpec(worldSpec: Readonly<WorldSpec>)` — return canonical two-space JSON with
  exactly one final newline.
- `formatDiagnostics(diagnostics: readonly Diagnostic[])` — render concise deterministic
  human-readable diagnostics.
- `Diagnostic`, `DiagnosticCode`, `DiagnosticSeverity`, `ValidationSuccess`, `ValidationFailure`,
  and `ValidationResult` — public diagnostic and result types.

A successful result is `{ valid: true, value: WorldSpec, diagnostics: readonly Diagnostic[] }`. A
failure is `{ valid: false, diagnostics: readonly Diagnostic[] }`. Callers should branch on the
`valid` discriminator rather than cast the original input.

Validation and parsing treat invalid documents as expected data outcomes. File I/O errors remain the
responsibility of the caller or CLI boundary. Normalization and serialization accept a typed,
already validated `WorldSpec`; use `parseWorldSpec` or `validateWorldSpec` first for unknown data.

## Deterministic normalization

`normalizeWorldSpec` returns a deep independent value and never mutates caller-owned arrays,
objects, or extension values.

The following arrays are treated as sets and sorted lexicographically:

- entity `tags`;
- provenance `referenceIds`;
- constraint `subjectIds`;
- constraint `targetIds`;
- lock `fieldPaths`; and
- budget `targetDevices`.

The schema requires each of these set-like arrays to contain unique values; normalization sorts but
does not silently repair duplicates.

The following top-level record collections are sorted by `id`:

- `entities`;
- `relationships`;
- `constraints`; and
- `locks`.

The following author-directed sequences preserve their original order:

- `references`;
- intent `mustHave`, `mustNotHave`, and `preferences`;
- style `architecture`, `shapeLanguage`, `materialFamilies`, `palette`, `lighting`, and
  `exclusions`; and
- all arrays nested inside open-ended `attributes` or `parameters` values.

Normalization does not assign IDs, repair invalid references, infer hierarchy, or resolve
constraints. It also does not use locale-sensitive comparison.

`stringifyWorldSpec` normalizes first and emits stable schema-defined object key order. Keys inside
open-ended JSON objects are sorted recursively; their array order is preserved. Serialization uses
two spaces, no trailing spaces, and exactly one `\n` after the closing brace. Repeated normalization
or serialization of equivalent input is byte-for-byte stable.

## CLI

Run the workspace CLI from the repository root through the root script:

```sh
pnpm worldspec <command>
```

### Validate a file

```sh
pnpm worldspec validate path/to/world.worldspec.json
```

Valid input writes `Valid WorldSpec: <file>` to standard output and exits `0`. Invalid JSON, schema
failure, or semantic failure writes a concise `Invalid WorldSpec: <file>` heading and formatted
diagnostics to standard error, then exits `1`.

For stable machine-readable output:

```sh
pnpm worldspec validate path/to/world.worldspec.json --json
```

JSON mode writes only one deterministic two-space-indented result to standard output:

```json
{
  "valid": true,
  "diagnostics": []
}
```

An invalid result uses `false` and includes structured diagnostic objects in the array. Automation
must use the diagnostic `code`, not parse a human message.

### Normalize to standard output

```sh
pnpm worldspec normalize path/to/world.worldspec.json
```

The command validates before emitting canonical JSON. It does not modify the input file.

### Normalize to an explicit file

```sh
pnpm worldspec normalize path/to/world.worldspec.json --output path/to/normalized.worldspec.json
```

Output is written only when `--output` is explicitly present. This is the only form that writes a
WorldSpec file; choose the path deliberately. Success writes
`Normalized WorldSpec written to <path>` to standard output.

### Print the schema

```sh
pnpm worldspec schema
```

This writes the deterministically generated draft 2020-12 schema to standard output.

### Exit codes

| Code | Meaning                                                                                                  |
| ---- | -------------------------------------------------------------------------------------------------------- |
| `0`  | The command succeeded; for `validate`, the input is valid.                                               |
| `1`  | The command was valid but the WorldSpec input was invalid JSON, schema-invalid, or semantically invalid. |
| `2`  | Usage, unsupported command or option, or file I/O error.                                                 |

Expected user errors do not print stack traces. Missing arguments, unknown commands, unsupported
flags, unreadable files, and output write failures use exit code `2`.

Top-level `--help` or `-h` prints usage and exits `0`:

```sh
pnpm worldspec --help
```

### Complete examples

```sh
# Validate the checked-in semantic mansion plan.
pnpm worldspec validate packages/worldspec/fixtures/valid/reference-mansion.worldspec.json

# Capture a script-friendly validation result.
pnpm worldspec validate candidate.worldspec.json --json > validation-result.json

# Review canonical output without changing a file.
pnpm worldspec normalize candidate.worldspec.json

# Write canonical output to a deliberate destination.
pnpm worldspec normalize candidate.worldspec.json --output candidate.normalized.worldspec.json

# Inspect the contract consumed by another language or tool.
pnpm worldspec schema > worldspec-0.1.0.schema.json
```

Shell redirection is shown only as a caller-selected destination. The CLI itself never performs a
network request.

## Schema generation and drift

TypeBox source in `packages/worldspec/src/schema.ts` is authoritative. The draft 2020-12 artifact at
`packages/worldspec/schema/worldspec-0.1.0.schema.json` is generated deterministically.

```sh
pnpm schema:generate
pnpm schema:check
```

Generation updates the artifact. The drift check regenerates the expected content in memory and
fails if it differs from the committed file. Re-running generation without a source-schema change
must create no diff.

## Versioning policy

Every document names its exact `schemaVersion`; consumers must reject versions they do not support
rather than guessing or silently coercing them.

- Documentation fixes and implementation corrections that do not change accepted shape or meaning
  may use a patch release.
- Backward-compatible contract additions require a new minor version and schema ID/artifact.
- Removed, renamed, narrowed, or reinterpreted contract elements require a major version.
- Each published version keeps its own schema artifact and documentation.
- Future migrations must be explicit, deterministic, intent-preserving, and tested. v0.1 does not
  provide migrations.

Because v0.1 domain objects are closed, adding even an optional domain property is a versioned
schema change.

## Known limits of v0.1

WorldSpec v0.1 intentionally does not contain or define:

- generated mesh data;
- Roblox Instance serialization;
- Luau scripts;
- material binaries;
- image data;
- arbitrary executable code;
- AI chain-of-thought;
- provider credentials;
- a spatial or architectural solver;
- constraint parameter semantics or constraint enforcement;
- compilation operations or transaction/rollback behavior;
- local-versus-world transform composition rules;
- relationship inference or geometric verification;
- lock enforcement, field-path wildcards, or mutation conflict resolution;
- reference fetching, trust, availability, or image understanding;
- performance measurement or budget enforcement;
- visual, traversal, gameplay, or architectural critique;
- migrations between schema versions; or
- Atlas, Forge, The Critic, a Roblox compiler, or another production integration.

The reference mansion fixture is deliberately a readable semantic plan, not a complete geometric
mansion and not proof that any future component has been implemented.
