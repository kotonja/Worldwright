# Roblox compiler v0.1 reference

## Status and scope

`@worldwright/roblox-compiler` version `0.1.0` is the Milestone 1 offline boundary between WorldSpec
`0.1.0` and any allowlisted Roblox adapter. It implements:

- strict per-entity Roblox directives;
- pure compilation to a canonical Roblox Manifest;
- validation and canonicalization of manifests and Scene Snapshots;
- deterministic dry-run planning to a Roblox Change Set;
- pure change-set simulation;
- a narrow asynchronous adapter contract;
- verified transaction execution with snapshot-based compensation; and
- a deterministic in-memory adapter for tests and local simulation.

This package does not connect to Roblox Studio, create live Instances, use Studio MCP, provide
Forge, emit Luau, or expose an `apply` CLI command. Milestone 3's separate
`@worldwright/studio-mcp-adapter` consumes its public contracts, `RobloxAdapter` interface, and
`applyRobloxChangeSet` executor for a bounded unsaved-sandbox integration; it does not change the
compiler wire contracts.

## Contract versions and schemas

All compiler domain objects are closed: unknown properties fail schema validation. The TypeBox
schemas are authoritative and generate deterministic draft 2020-12 JSON Schema artifacts.

| Contract              | Version | Schema ID                                 | Source                    | Generated artifact                           |
| --------------------- | ------- | ----------------------------------------- | ------------------------- | -------------------------------------------- |
| Roblox directive      | `0.1.0` | `urn:worldwright:roblox-directive:0.1.0`  | `src/directive-schema.ts` | `schema/roblox-directive-0.1.0.schema.json`  |
| Roblox Manifest       | `0.1.0` | `urn:worldwright:roblox-manifest:0.1.0`   | `src/contract-schema.ts`  | `schema/roblox-manifest-0.1.0.schema.json`   |
| Roblox Scene Snapshot | `0.1.0` | `urn:worldwright:roblox-snapshot:0.1.0`   | `src/contract-schema.ts`  | `schema/roblox-snapshot-0.1.0.schema.json`   |
| Roblox Change Set     | `0.1.0` | `urn:worldwright:roblox-change-set:0.1.0` | `src/contract-schema.ts`  | `schema/roblox-change-set-0.1.0.schema.json` |

The compiler version and all four contract versions are `0.1.0`. The compiler accepts only WorldSpec
schema version `0.1.0`.

Managed IDs use the WorldSpec identifier rules: 1 through 128 characters matching
`^[a-z][a-z0-9]*(?:-[a-z0-9]+)*$`. SHA-256 values are exactly 64 lowercase hexadecimal characters.
All numbers must be finite JSON numbers. Counts are non-negative safe integers no greater than
`9,007,199,254,740,991`.

## Roblox directive schema

The compiler reads exactly one directive from every entity at:

```ts
entity.attributes['worldwright.roblox'];
```

The directive is a strict class-discriminated union. The compiler never guesses a class and never
silently skips an entity.

### Container directive

```json
{
  "schemaVersion": "0.1.0",
  "mode": "container",
  "className": "Model"
}
```

`className` must be `Folder` or `Model`. No other properties are accepted.

### Part directive

```json
{
  "schemaVersion": "0.1.0",
  "mode": "primitive",
  "className": "Part",
  "shape": "Block",
  "material": "Concrete",
  "color": {
    "r": 118,
    "g": 124,
    "b": 132
  },
  "transparency": 0,
  "canCollide": true,
  "canQuery": true,
  "canTouch": true,
  "castShadow": true
}
```

`Part` requires `shape`, which must be `Block`, `Ball`, or `Cylinder`.

### Wedge directives

`WedgePart` and `CornerWedgePart` use the same primitive visual and collision fields but forbid
`shape`; their class determines geometry.

```json
{
  "schemaVersion": "0.1.0",
  "mode": "primitive",
  "className": "WedgePart",
  "material": "Slate",
  "color": {
    "r": 86,
    "g": 91,
    "b": 100
  },
  "transparency": 0,
  "canCollide": true,
  "canQuery": true,
  "canTouch": false,
  "castShadow": true
}
```

For every primitive, `r`, `g`, and `b` are integers from 0 through 255; `transparency` is from 0
through 1; and all four collision/shadow fields are booleans. Primitives are always anchored, so
anchoring is not a directive option.

### Directive semantic rules

- The root WorldSpec entity must use a container directive.
- Container entities may contain containers and primitives.
- A primitive must provide both `transform` and `bounds` and must be a leaf in the WorldSpec parent
  graph.
- `Part` requires `shape`; wedge classes forbid it.
- A missing directive is an error, as is an invalid or unknown directive property.
- The compiler does not copy the remaining open WorldSpec attributes into managed Roblox state.

The directive lives in WorldSpec's existing open attributes map. It does not add a field to or
change the WorldSpec `0.1.0` wire schema.

## Allowlists

### Classes

| Mode      | Allowed class     | Geometry selector                                 |
| --------- | ----------------- | ------------------------------------------------- |
| Container | `Folder`          | None; properties are an empty closed object.      |
| Container | `Model`           | None; properties are an empty closed object.      |
| Primitive | `Part`            | Required `shape`: `Block`, `Ball`, or `Cylinder`. |
| Primitive | `WedgePart`       | Class-defined; `shape` is forbidden.              |
| Primitive | `CornerWedgePart` | Class-defined; `shape` is forbidden.              |

No other Roblox class is valid. In particular, the compiler cannot represent scripts, `MeshPart`,
`UnionOperation`, `Terrain`, decals, textures, or surface appearances.

### Primitive properties

The closed primitive property record contains only:

| Property                  | Type and rule                                                        |
| ------------------------- | -------------------------------------------------------------------- |
| `position`                | Finite `{ x, y, z }` world-space vector.                             |
| `rotationEulerDegreesXYZ` | Finite `{ x, y, z }` world-space Euler angles in degrees, XYZ order. |
| `size`                    | Finite `{ x, y, z }`; every component is greater than zero.          |
| `anchored`                | Literal `true`.                                                      |
| `material`                | One value from the material allowlist.                               |
| `color`                   | `{ r, g, b }` with integer channels from 0 through 255.              |
| `transparency`            | Finite number from 0 through 1.                                      |
| `canCollide`              | Boolean.                                                             |
| `canQuery`                | Boolean.                                                             |
| `canTouch`                | Boolean.                                                             |
| `castShadow`              | Boolean.                                                             |
| `shape`                   | `Part` only; `Block`, `Ball`, or `Cylinder`.                         |

Container properties are an empty closed object. Arbitrary Roblox properties, physical properties,
collision-group creation, unanchored primitives, asset IDs, and content URLs are not supported.

## Spatial compilation semantics

Compiler v0.1 uses world-space transforms. Parent hierarchy controls organization only and does not
compose transforms. A container's WorldSpec transform or bounds remains semantic source data and
does not move descendants.

For a primitive:

```text
position = transform.position
rotationEulerDegreesXYZ = transform.rotationEulerDegrees
size.x = bounds.size.x * transform.scale.x
size.y = bounds.size.y * transform.scale.y
size.z = bounds.size.z * transform.scale.z
anchored = true
```

Every final size component must be positive. Compilation reports an error rather than emitting an
invalid size. The calculation does not mutate the WorldSpec input and does not use locale-sensitive
number formatting.

A live adapter must construct rotation equivalent to:

```lua
CFrame.new(position)
    * CFrame.fromEulerAnglesXYZ(
        math.rad(x),
        math.rad(y),
        math.rad(z)
    )
```

That mapping documents the contract for a future adapter; no Luau is generated or executed in
Milestone 1.

## Managed node schema

Manifest and snapshot managed nodes share one strict class-discriminated representation:

```ts
type RobloxManagedNode = {
  id: string;
  entityKind: RobloxEntityKind;
  name: string;
  parentId?: string;
  className: 'Folder' | 'Model' | 'Part' | 'WedgePart' | 'CornerWedgePart';
  properties: RobloxContainerProperties | RobloxPrimitiveProperties;
  attributes: RobloxManagedAttributes;
};
```

`entityKind` is one of the 13 WorldSpec entity kinds: `world`, `region`, `district`, `parcel`,
`structure`, `floor`, `room`, `route`, `terrain`, `landmark`, `object`, `spawn`, or `interaction`.
The `properties` branch must agree with `className` exactly.

### Managed attribute mapping

The closed attribute record is:

| Attribute                    | Rule                                                                                                                                            |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `WorldwrightManaged`         | Always `true`.                                                                                                                                  |
| `WorldwrightProjectId`       | Equals the manifest or snapshot project ID.                                                                                                     |
| `WorldwrightEntityId`        | Equals the node `id`.                                                                                                                           |
| `WorldwrightEntityKind`      | Equals the node `entityKind`.                                                                                                                   |
| `WorldwrightCompilerVersion` | Literal `0.1.0`.                                                                                                                                |
| `WorldwrightSourceHash`      | Required on the root and omitted on non-root nodes. In a manifest it equals `source.worldSpecHash`; a snapshot records the observed root value. |

Managed identity relies on IDs and attributes, never `name`. Names are display values. Reference
URIs, prompts, notes, provenance details, private source data, and arbitrary WorldSpec attributes
are not copied into this record.

## Roblox Manifest schema

The manifest is the complete desired managed state:

```ts
interface RobloxManifest {
  schemaVersion: '0.1.0';
  compilerVersion: '0.1.0';
  source: {
    worldSpecSchemaVersion: '0.1.0';
    projectId: string;
    worldSpecHash: string;
  };
  target: { service: 'Workspace' };
  rootNodeId: string;
  nodes: RobloxManagedNode[];
  measurements: {
    instances: number;
    containers: number;
    primitives: number;
  };
}
```

`nodes` contains at least one node. Arbitrary services and service paths are not valid targets.

### Manifest invariants

- Node IDs are unique and every parent ID resolves.
- The managed parent graph is acyclic and every node reaches `rootNodeId`.
- `rootNodeId` equals the source WorldSpec `rootEntityId`.
- The root has no `parentId`, is a `Folder` or `Model`, and carries `WorldwrightSourceHash`.
- Every non-root node has a parent and omits `WorldwrightSourceHash`.
- The hierarchy mirrors WorldSpec `parentId` exactly.
- Every WorldSpec entity maps to exactly one node with the same ID, kind, and name.
- Node attributes agree with the source, node identity, and compiler version.
- Primitive properties agree with the class-specific allowlist and final size is positive.
- `measurements.instances` equals `nodes.length`.
- `measurements.containers + measurements.primitives` equals `measurements.instances` and each count
  agrees with node classes.

`source.worldSpecHash` is the lowercase SHA-256 of `stringifyWorldSpec(validatedWorldSpec)`. It is
not a hash of the caller's original, potentially non-canonical representation.

### Budget behavior

If `budgets.limits.instances` exists and the manifest would exceed it, compilation fails with
`compiler.instance_budget_exceeded`.

The offline v0.1 compiler cannot measure engine-rendered triangles or texture memory. Supplying
either corresponding WorldSpec limit does not fail otherwise valid compilation, but produces a
`compiler.budget_not_evaluated` warning. The warning means the budget was not evaluated; it is not a
claim that the budget passed.

## Roblox Scene Snapshot schema

A snapshot is observed state for one managed project in `Workspace`:

```ts
interface RobloxSnapshot {
  schemaVersion: '0.1.0';
  projectId: string;
  target: { service: 'Workspace' };
  rootNodeId?: string;
  nodes: RobloxManagedNode[];
  unmanagedRoots: Array<{
    snapshotId: string;
    parentNodeId: string;
    name: string;
  }>;
}
```

An empty snapshot is valid and omits `rootNodeId`:

```json
{
  "schemaVersion": "0.1.0",
  "projectId": "project-primitive-courtyard",
  "target": {
    "service": "Workspace"
  },
  "nodes": [],
  "unmanagedRoots": []
}
```

An empty snapshot is valid only as an observation. It cannot be the desired result of a successful
forward change set because every Roblox Manifest is non-empty and has one managed root.

### Snapshot invariants

- Managed node IDs are unique.
- An empty snapshot has no root and no unmanaged roots.
- A non-empty snapshot declares one root; that root exists, has no parent, and every node reaches
  it.
- Parent IDs resolve and the managed hierarchy is acyclic.
- All managed metadata agrees with `projectId`, identity, kind, and compiler version.
- Only supported classes and matching strict property records exist.
- Every non-empty snapshot root carries the observed `WorldwrightSourceHash`; snapshots have no
  independent `source` record with which to prove provenance equality.
- `WorldwrightSourceHash` is absent on every non-root snapshot node.
- Each unmanaged root references an existing managed `parentNodeId`.
- Unmanaged `snapshotId` values are opaque, stable, non-empty strings of at most 128 characters and
  are unique within the snapshot.

### Meaning of `unmanagedRoots`

An unmanaged-root record says: user-owned content exists as a direct child below this managed parent
and must be protected. `snapshotId` is observation identity only; it is not a WorldSpec ID or
managed-node ID. The full unmanaged subtree is intentionally not serialized.

Before planning a delete or reparent, the planner checks the affected managed node and all managed
descendants. Any unmanaged-root marker in that region produces `plan.unmanaged_descendant_conflict`.
Worldwright does not delete, move, clone, quarantine, inspect, or mutate unmanaged content. A
property-only update remains eligible if it does not destroy or reparent that content.

The complete snapshot hash includes all unmanaged-root records. Adding or changing user-owned
content therefore invalidates an already planned change set.

## Roblox Change Set schema

A change set is a dry-run transition from one exact snapshot to one exact manifest result:

```ts
interface RobloxChangeSet {
  schemaVersion: '0.1.0';
  compilerVersion: '0.1.0';
  preconditions: {
    projectId: string;
    target: { service: 'Workspace' };
    baseSnapshotHash: string;
    desiredManifestHash: string;
    resultSnapshotHash: string;
  };
  operations: RobloxChangeOperation[];
  summary: {
    creates: number;
    updates: number;
    deletes: number;
    total: number;
  };
}
```

Every hash is a 64-character lowercase SHA-256 value. The preconditions protect separate claims:

- `baseSnapshotHash` binds the complete observed boundary before mutation, including unmanaged-root
  records;
- `desiredManifestHash` binds the simulated managed result to the exact desired manifest and is an
  enforced provenance and integrity link, not descriptive metadata; and
- `resultSnapshotHash` independently binds the complete expected result snapshot, including its
  preserved unmanaged-root records.

`summary.total` equals the operation count and the sum of the three phase counts.

### Operation union

Create contains the complete desired node:

```ts
interface RobloxCreateOperation {
  id: `create:${string}`;
  type: 'create';
  node: RobloxManagedNode;
}
```

Update contains complete before and after nodes:

```ts
interface RobloxUpdateOperation {
  id: `update:${string}`;
  type: 'update';
  before: RobloxManagedNode;
  after: RobloxManagedNode;
}
```

Delete contains the complete before node:

```ts
interface RobloxDeleteOperation {
  id: `delete:${string}`;
  type: 'delete';
  before: RobloxManagedNode;
}
```

The operation ID is exactly `<type>:<nodeId>`, where `nodeId` follows the managed identifier rules.
Update before and after IDs match, and their class names must match. An equal before and after value
is invalid. Class changes fail with `plan.class_change_unsupported` instead of becoming an implicit
delete and create.

### Deterministic operation order

Operations are emitted and executed in three safe phases:

1. Creates, ancestors before descendants.
2. Updates and reparents, after required parents exist.
3. Deletes, descendants before ancestors.

Within an equivalent phase and hierarchy depth, operations sort by node ID using code-point order.
The order is part of the change-set contract; consumers must not reorder it.

## Dry-run planning

```ts
planRobloxChangeSet(
  currentSnapshot: unknown,
  desiredManifest: unknown,
): PlanResult
```

Planning validates both inputs, rejects project or target mismatches, and never mutates inputs or
performs I/O. It returns a normalized change set and the exact expected result snapshot. That
snapshot preserves the base snapshot's `unmanagedRoots` unchanged.

A snapshot already equal to desired state produces a valid no-op change set with zero operations and
zero summary counts. Planning the same normalized inputs repeatedly produces byte-identical output.

Planning rejects:

- invalid manifests or snapshots;
- project or target mismatch;
- a class change under a stable node ID;
- an unsupported root transition; and
- deletion or reparenting across a managed lineage that contains unmanaged roots.

These are visible conflicts. The planner never silently drops a desired change and never modifies a
different project.

## Pure simulation

```ts
simulateRobloxChangeSet(
  snapshot: unknown,
  changeSet: unknown,
): SimulationResult
```

Simulation:

1. validates both contracts;
2. verifies project, target, and complete base snapshot hash;
3. uses the same deterministic ownership analysis as planning to reject a protected delete or
   reparent before a generic graph error can obscure the unmanaged-content cause;
4. enforces each operation's complete before-state precondition;
5. enforces operation order, parent existence, acyclic hierarchy, and identity;
6. applies operations to an independent value without mutating caller data;
7. preserves unmanaged roots;
8. validates the result snapshot and derives its corresponding desired manifest;
9. hashes that manifest and compares it with `desiredManifestHash`; and
10. independently hashes the complete result snapshot and compares it with `resultSnapshotHash`.

Desired-manifest derivation accepts only a valid, non-empty result snapshot with exactly one valid
root carrying `WorldwrightSourceHash`. It constructs:

- current Manifest `schemaVersion` and compiler version;
- supported WorldSpec schema version;
- `source.projectId` from snapshot `projectId`;
- `source.worldSpecHash` only from the root's validated `WorldwrightSourceHash`;
- `target`, `rootNodeId`, and managed `nodes` from the snapshot; and
- instance, container, and primitive measurements recomputed exactly from node classes.

The derived value must itself pass Manifest validation. Derivation is deterministic, does not mutate
the snapshot, excludes `unmanagedRoots` because they are observed ownership evidence rather than
desired state, and does not invent or accept source metadata from another field. Repeated derivation
and canonical hashing are byte-identical.

`simulation.desired_manifest_invalid` reports that the simulated managed result cannot form a valid
non-empty manifest. This rejects a hand-authored delete-all transition. A valid derived manifest
whose hash differs from the precondition reports `simulation.desired_manifest_hash_mismatch`. A
caller therefore cannot tamper with an operation and make it valid merely by recomputing
`resultSnapshotHash`; both hashes must independently match.

For ownership conflicts, `simulation.unmanaged_descendant_conflict` points to the responsible
operation: `/operations/{index}/before` for deletion and `/operations/{index}/after/parentId` for
reparenting. Its `relatedId` is the minimum Unicode-code-point unmanaged `snapshotId` in the
protected subtree. An unrelated property-only update remains valid, and simulation leaves all
unmanaged records unchanged.

Simulation performs no external I/O. Transaction execution uses it as a complete preflight before
the first mutation call.

## Adapter and transaction behavior

The adapter surface is scoped and allowlisted:

```ts
interface RobloxAdapterScope {
  projectId: string;
  target: { service: 'Workspace' };
}

interface RobloxAdapter {
  readSnapshot(scope: RobloxAdapterScope): Promise<unknown>;
  prepareForMutation?(
    input: RobloxMutationPreparationInput,
  ): Promise<{ success: true } | { success: false; diagnostics: readonly RobloxDiagnostic[] }>;
  createNode(scope: RobloxAdapterScope, node: RobloxManagedNode): Promise<void>;
  updateNode(
    scope: RobloxAdapterScope,
    before: RobloxManagedNode,
    after: RobloxManagedNode,
  ): Promise<void>;
  deleteNode(scope: RobloxAdapterScope, before: RobloxManagedNode): Promise<void>;
}
```

There is no raw property setter, arbitrary class creation method, or script method. Each adapter is
responsible for faithfully implementing the fixed class and property allowlists within the selected
project and target.

`applyRobloxChangeSet(adapter, changeSet)` performs these stages:

1. Validate the change set.
2. Read, validate, normalize, and hash the current snapshot.
3. Reject a stale base hash before any mutation.
4. Simulate the complete change set, enforcing both `desiredManifestHash` and `resultSnapshotHash`.
5. If there are zero operations, return `noop` success immediately with the verified preflight
   snapshot and hash.
6. If the adapter exposes the optional generic preparation hook, call it with detached frozen
   normalized transaction values.
7. After successful preparation, read a fresh complete snapshot and require its hash to equal the
   exact initial base hash.
8. Apply operations sequentially, or through exact deterministic batches in the additive batched
   executor.
9. Read and validate final state.
10. Return `applied` success only when the complete final hash equals `resultSnapshotHash`.

The no-op path reads the adapter exactly once. It performs zero mutations, does not perform a second
read, and never enters rollback. It returns `success: true`, `status: "noop"`, the verified
preflight snapshot, empty diagnostics, `operationsAttempted: 0`, and equal initial and final
snapshot hashes. A preflight failure, including a desired-manifest mismatch or delete-all result,
likewise performs zero mutation and does not attempt rollback.

The optional preparation hook exists for an adapter-owned privileged boundary that must be
established after pure authorization checks but before mutation. It returns no identity value to the
compiler and cannot change the Change Set authorization unit. Adapters without the hook retain the
original read and mutation sequence. A preparation failure, an invalid post-preparation snapshot, or
a changed post-preparation base reports zero attempted node operations and does not enter rollback.
Successful preparation always requires the second exact-base observation before batch planning or
the first mutation call. Any private adapter preparation state remains adapter-owned and must not be
added to compiler snapshots, hashes, results, or diagnostics.

If application or verification fails, the executor records the valid observable failure state and
checks rollback admissibility before any compensating mutation. It compares a fresh normalized
snapshot with the initial snapshot and only the prefix of forward operations whose adapter calls
were attempted:

- every managed node not targeted by that prefix must exactly equal the initial node, and an
  initially absent unrelated ID must remain absent;
- an attempted create target may be absent or exactly equal the normalized created node;
- an attempted update target may exactly equal its normalized `before` or `after` node;
- an attempted delete target may exactly equal its normalized `before` node or be absent;
- the complete normalized `unmanagedRoots` collection must exactly equal the initial collection; and
- root presence or identity may differ only when explained by an attempted root create or delete.

An unrelated managed property change, addition, or deletion; any changed unmanaged-root record; or a
third state at an attempted target makes compensation unsafe. The executor reports
`transaction.rollback_unsafe_observed_state`, calls no compensating mutation method, and preserves
the observed state. Its rollback result is attempted and failed even though the guard permits zero
compensating calls. It reports the valid pre-compensation hash as `observedFailureSnapshotHash` and
omits `observedAfterRollbackSnapshotHash` because no compensating mutation or post-attempt
observation occurred. If the observation is admissible, it plans a transition back to the exact
initial snapshot, applies it, reads state again, and compares the complete hash with the original
initial hash. It reports rollback success only after that final verification. A rollback can fail
and is reported explicitly.

### Concurrency model

`baseSnapshotHash` protects the complete boundary before mutation. Complete `before` node checks
protect targeted operations. The rollback admissibility guard prevents compensation from overwriting
detected unrelated changes.

Worldwright does not claim engine-level atomicity or transaction isolation. A live adapter must
serialize Worldwright transactions within one project scope and treat creator edits during an
asynchronous transaction as a concurrency hazard. When causality is uncertain, Worldwright reports
rollback failure instead of overwriting unrelated state. ChangeHistoryService may support future
creator undo, but it does not provide transaction isolation and does not replace these checks. The
Milestone 3 Studio MCP adapter does not call ChangeHistoryService.

The package's testing subpath exports the deterministic in-memory adapter and fault-injection
helpers. The test adapter can model a failure before an operation, a failure after mutation,
incorrect post-apply state, and rollback failure without clocks, sleeps, races, or random behavior.
It is not a production or Studio adapter.

## Result types

Expected invalid input, planning conflicts, stale state, adapter failures, and verification failures
are discriminated data outcomes. Normal callers do not need to catch them or parse stack traces.

```ts
type CompileResult =
  | { success: true; manifest: RobloxManifest; diagnostics: RobloxDiagnostic[] }
  | { success: false; diagnostics: RobloxDiagnostic[] };

type PlanResult =
  | {
      success: true;
      changeSet: RobloxChangeSet;
      expectedSnapshot: RobloxSnapshot;
      diagnostics: RobloxDiagnostic[];
    }
  | { success: false; diagnostics: RobloxDiagnostic[] };

type SimulationResult =
  | { success: true; snapshot: RobloxSnapshot; diagnostics: RobloxDiagnostic[] }
  | { success: false; diagnostics: RobloxDiagnostic[] };
```

Contract validators use a separate `valid` discriminator and return an independent normalized value
only on success.

Apply success includes:

- `success: true`;
- `status: "applied" | "noop"`;
- the verified resulting `snapshot`;
- diagnostics;
- `operationsAttempted`;
- `initialSnapshotHash`; and
- `finalSnapshotHash`.

Apply failure includes:

- `success: false`;
- `stage`, one of `change-set-validation`, `snapshot-read`, `snapshot-validation`, `stale-check`,
  `preflight`, `apply`, or `verification`;
- diagnostics;
- `operationsAttempted`;
- a discriminated `rollback` result;
- optional `initialSnapshotHash`; and
- optional `observedFailureSnapshotHash`.

`observedFailureSnapshotHash` has exactly one meaning: the valid partial or incorrect state observed
before compensation. It is omitted if that observation is unreadable or invalid and is never
overwritten with a restored hash.

Rollback results distinguish:

- not attempted;
- attempted and verified successful, with `restoredSnapshotHash` equal to `initialSnapshotHash`; and
- attempted but failed, with diagnostics and optional `observedAfterRollbackSnapshotHash`, the
  latest valid state observed after the failed compensation attempt.

No field changes meaning by outcome. A no-op success contains only success hashes and no rollback
result.

## Diagnostics

Every diagnostic has stable machine-readable shape:

```ts
interface RobloxDiagnostic {
  code: RobloxDiagnosticCode;
  severity: 'error' | 'warning';
  path: string;
  message: string;
  relatedId?: string;
}
```

`path` is a JSON Pointer-like location; the empty document root is rendered as `/` in human output.
Automation must branch on `code`, never parse `message`. Ordering is deterministic by path, code,
severity, message, and related ID.

### Diagnostic code catalog

| Code                                         | Meaning                                                                          |
| -------------------------------------------- | -------------------------------------------------------------------------------- |
| `json.invalid`                               | A value is not compatible with the JSON data model.                              |
| `contract.schema_invalid`                    | A manifest, snapshot, or change set violates its closed schema.                  |
| `contract.id_duplicate`                      | A contract reuses an ID that must be unique.                                     |
| `contract.parent_missing`                    | A required managed parent does not exist.                                        |
| `contract.parent_cycle`                      | The managed parent graph contains a cycle.                                       |
| `contract.unreachable`                       | A managed node does not reach the selected root.                                 |
| `contract.root_invalid`                      | Root presence, class, parentage, or metadata is invalid.                         |
| `contract.metadata_invalid`                  | Project, identity, compiler, source-hash, or unmanaged-root metadata disagrees.  |

compiler-level transaction diagnostics. Their sanitized message names the underlying stable
diagnostic code, so malformed operations and desired-manifest mismatches remain actionable without
exposing raw Ajv prose, hostile adapter messages, or stack traces. Wrapped diagnostics are sorted
deterministically.

## Canonical normalization and hashing

Directive serialization where needed, manifests, snapshots, change sets, diagnostics, and CLI JSON
output are deterministic:

- every object key is recursively ordered by Unicode code point;
- node and unmanaged-root collections use contract-specific code-point keys;
- operation arrays retain deterministic safe execution order;
- set-like arrays are sorted without `localeCompare`;
- JSON uses two spaces and exactly one final `\n`;
- generated artifacts contain no timestamps, random UUIDs, absolute machine paths, or
  platform-dependent line endings; and
- hashes are lowercase hexadecimal SHA-256 over the complete canonical value.

Intentional functions hash manifests, complete snapshots, managed snapshot state where useful, and
change sets. Hash fields are not embedded into the object they hash, avoiding self-reference.

## CLI

Run the source CLI from the repository root:

```sh
pnpm roblox-compiler <command>
```

### Compile

```sh
pnpm roblox-compiler compile <worldspec-file>
pnpm roblox-compiler compile <worldspec-file> --output <manifest-file>
pnpm roblox-compiler compile <worldspec-file> --json
```

Without `--output`, human mode writes the canonical manifest to standard output. An output file is
written only when its path is explicit; the CLI never overwrites the input implicitly. JSON mode
emits one stable machine-readable result including warnings and diagnostics.

### Plan

```sh
pnpm roblox-compiler plan <manifest-file>
pnpm roblox-compiler plan <manifest-file> --snapshot <snapshot-file>
pnpm roblox-compiler plan <manifest-file> --snapshot <snapshot-file> --output <change-set-file>
pnpm roblox-compiler plan <manifest-file> --snapshot <snapshot-file> --json
```

When `--snapshot` is omitted, planning uses a canonical empty snapshot for the manifest's project
and `Workspace` target. The command emits or writes the canonical dry-run change set; it does not
apply it.

### Exit codes

| Code | Meaning                                                                                        |
| ---- | ---------------------------------------------------------------------------------------------- |
| `0`  | Compile or plan succeeded, including warnings or a no-op plan.                                 |
| `1`  | The command was valid but input was invalid, compilation failed, or planning found a conflict. |
| `2`  | Usage, unsupported option or command, or file I/O failed.                                      |

Expected errors do not print stack traces. The CLI performs no shell composition or network call.
The emitted `dist/cli.js` is smoke-tested through `process.execPath` for documented output and exit
paths.

There is deliberately no CLI command that claims to apply a change set to Roblox Studio.

## Security and ownership boundaries

Compiler and transaction enforcement includes:

- closed class, property, material, attribute, and operation allowlists;
- no executable source, dynamic evaluation, asset IDs, content URLs, or arbitrary attribute copy;
- no network, AI provider, database, authentication, telemetry, or analytics;
- project and `Workspace` target scoping on every adapter operation;
- no managed identity based on display name;
- no mutation of a different project or unmanaged node;
- no delete or reparent across unmanaged descendants;
- no stale change-set application;
- no simulated result whose reconstructed Manifest disagrees with `desiredManifestHash`;
- no compensation when observed changes fall outside the attempted-operation envelope;
- deterministic operation IDs, output, hashes, and diagnostics; and
- verified final state or explicitly reported verified/failed compensation.

The in-memory adapter demonstrates the protocol without broadening its authority. The separately
authorized Studio MCP adapter introduced in Milestone 3 preserves these boundaries while adding an
unsaved-local-place, stopped-Edit-mode engine mapping. Forge, published-place authorization, and
other transports still require explicit milestones.

## Limitations

Compiler v0.1 intentionally does not provide:

- architectural reasoning, spatial planning, constraint solving, or relationship realization;
- local-space parent transform composition;
- one-to-many entity expansion or class migration;
- arbitrary Roblox classes or properties;
- meshes, unions, terrain, decals, textures, surface appearances, asset insertion, or generated
  materials;
- scripts, gameplay logic, Rojo, roblox-ts, or Lune;
- collision-group creation or custom physical properties;
- reliable engine-rendered triangle or texture-memory measurement;
- serialization or repair of unmanaged subtrees;
- a live Roblox Studio connection inside this compiler package, Forge interface, plugin, or
  ChangeHistoryService integration; the separate Studio MCP adapter supplies only the bounded
  Milestone 3 live boundary;
- visual, gameplay, traversal, or performance observation through The Critic; or
- production services, deployment, authentication, telemetry, or analytics.

Container transforms remain semantic only, and all primitive transforms are interpreted in world
space. These limitations are contract behavior, not implied future capability.

## Versioning policy

Every directive, manifest, snapshot, and change set names its exact `schemaVersion`. Consumers must
reject unsupported versions rather than guess or silently coerce them.

- Documentation corrections and implementation fixes that preserve accepted shape and meaning may
  use a patch release.
- Backward-compatible additions require a new minor contract version and generated schema artifact.
- Removed, renamed, narrowed, or reinterpreted fields require a major version.
- Changes to spatial semantics, managed ownership, operation ordering, hash inputs, or transaction
  guarantees are semantic contract changes and follow the same policy.
- Each published version retains its own schema artifacts and documentation.
- Future migrations must be explicit, deterministic, intent-preserving, and tested. v0.1 provides no
  migration protocol.

Compiler `0.1.0` is the accepted Milestone 1 contract. The pre-merge director-review correction
replaced its ambiguous transaction result-hash wording before acceptance without changing the
directive, Manifest, Snapshot, or Change Set JSON wire schemas and without introducing a migration.
These compatibility rules govern subsequent changes.

The compiler and WorldSpec versions are independent. This compiler supports WorldSpec `0.1.0`
without modifying its wire contract.
