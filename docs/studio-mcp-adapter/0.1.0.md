# Studio MCP Adapter 0.1 reference

## Status and purpose

`@worldwright/studio-mcp-adapter` version `0.1.0` is Worldwright's first live Roblox side-effect
boundary. It connects to the built-in Roblox Studio MCP server over local stdio, validates its tool
surface, selects one exact Studio session, observes Worldwright-managed Instances as an existing
Roblox Scene Snapshot, and implements the existing compiler `RobloxAdapter` interface with fixed
Edit-mode bridge actions.

The package applies an already validated Roblox Change Set through the compiler's public
`applyRobloxChangeSet` transaction executor. It does not define a replacement Manifest, Snapshot,
Change Set, simulator, stale-state check, or rollback model.

Version `0.1.0` is restricted to a new unsaved local Studio place with both `game.PlaceId` and
`game.GameId` equal to zero and the data model stopped in Edit mode. It is not a general Studio
automation API and cannot expose arbitrary Luau through its public package or CLI.

The root package exposes safe session listing, read-only or exact-selected adapter connection,
contract schemas and validation, deterministic receipt construction, and viewport evidence hashing.
It does not export the raw MCP client, protocol factory, generic tool invocation, fixed-program
issuer, or evidence file writer. Fake protocols and the settled post-mutation fault wrapper are
available only from the explicit `@worldwright/studio-mcp-adapter/testing` subpath.

## Versions, schemas, and limits

The adapter, fixed bridge protocol, and receipt each declare `0.1.0`:

| Contract               | Schema ID                                      |
| ---------------------- | ---------------------------------------------- |
| Studio bridge request  | `urn:worldwright:studio-bridge-request:0.1.0`  |
| Studio bridge response | `urn:worldwright:studio-bridge-response:0.1.0` |
| Studio Apply Receipt   | `urn:worldwright:studio-apply-receipt:0.1.0`   |

TypeBox source schemas are canonical. Checked-in draft 2020-12 JSON Schemas are generated
deterministically and drift-checked. Domain objects are closed with `additionalProperties: false`.
Validation rejects non-JSON values before schema evaluation, including functions, symbols, class
instances, dates, big integers, `undefined`, non-finite numbers, sparse arrays, accessors,
non-enumerable custom properties, and cycles.

| Resource                                   |   Limit |
| ------------------------------------------ | ------: |
| Operations in one Studio change set        |     512 |
| Managed nodes in one selected project      |   2,048 |
| Instances in one Workspace structural scan |  65,536 |
| Encoded adapter state for one node         | 256 KiB |
| Encoded bridge payload                     |   4 MiB |
| Complete framed bridge output              |  96 KiB |
| General MCP tool result validation         |  16 MiB |
| Decoded JPEG file bytes                    |  16 MiB |
| Declared JPEG pixel samples                | 256 MiB |
| MCP content items                          |       8 |
| Receipt diagnostics                        |   4,099 |
| Process startup                            |    15 s |
| Session registration discovery             |     6 s |
| Tool call                                  |    30 s |
| Process close                              |     7 s |
| Engine numeric comparison epsilon          | 0.00001 |

Exceeding a bound fails before the value is trusted or mutated. JPEG parsing also caps marker and
scan counts plus the frame dimensions implied by its component count.

## Command resolution

Production uses the documented local stdio server only.

On Windows the resolver reads `process.env.LOCALAPPDATA`, validates it, expands the installation
path in Node, and returns:

```text
command: cmd.exe
args: /d /s /c call <expanded LOCALAPPDATA>\Roblox\mcp.bat
```

It does not pass a literal `%LOCALAPPDATA%` token, use `shell: true`, or expose the expanded local
path in a diagnostic. On macOS the executable is:

```text
/Applications/RobloxStudio.app/Contents/MacOS/StudioMCP
```

Linux is unsupported because Roblox Studio is unsupported there. The resolver fails rather than
guessing. The public package and CLI accept no custom MCP command, URL, transport, registry server,
or downloaded executable. A protocol-injection factory exists only on the package testing subpath;
production does not support Streamable HTTP, SSE, TCP, remote MCP, or `npx` startup.

## Client lifecycle and MCP results

The client uses `@modelcontextprotocol/sdk` `1.29.0` through a narrow wrapper. Startup, session
registration discovery, and each tool call have bounded timeouts. Studio can register open sessions
asynchronously after the stdio handshake, so discovery retries empty or transiently partial lists
for at most six seconds. On Windows, close uses the SDK-owned `cmd.exe` PID with the system
`taskkill.exe /T /F`, verifies that PID exited, and then completes SDK shutdown. This prevents the
batch-launched `StudioMCP.exe` child from surviving timeout poisoning. The seven-second outer close
bound exceeds the SDK's internal shutdown stages. Failures do not leave interactive stdin or forward
an uncontrolled stderr stream.

Every response is validated. The built-in MCP has been observed to impose an approximately
100,000-byte ceiling on tool text, so every complete framed bridge output is capped conservatively
at 96 KiB (98,304 bytes), including its prefix and final newline. The existing 16 MiB bound remains
the general outer validation ceiling for MCP results and capture framing; it is not the snapshot
transport budget. Viewport capture accepts only one exact lowercase `image/jpeg` item with canonical
base64. The encoded base64, media type, and item type all count against the unchanged 16 MiB outer
result limit before decoding, so that outer limit constrains an image to slightly less than 12 MiB
of JPEG file bytes even though the independent decoded-file limit remains 16 MiB.

The JPEG validator accepts one standalone 8-bit baseline-sequential SOF0 frame with positive
dimensions, one complete scan, and exactly 1 or 3 components. It bounds marker traversal and
declared pixel samples; validates segment lengths, nonzero 8-bit quantization tables, bounded
Huffman tables, table references, component selectors, byte stuffing, and ordered restart markers;
and requires exact SOI and terminal EOI with no trailing bytes. Progressive, arithmetic,
hierarchical, multi-scan, abbreviated-table, malformed, and ambiguous streams fail closed. It does
not decode pixels, so a capture is structural and hash evidence rather than visual-quality proof.
Unsupported or ambiguous mixed MCP content also fails closed. Results and errors do not reveal raw
MCP payloads, Studio stack traces, server stderr, usernames, environment values, or installation
paths. A rejected or timed-out privileged `execute_luau` call poisons and closes that uncertain
transport lane; compensation is never reported through it.

## Required and optional tools

The handshake requires and schema-checks these exact tools:

| Tool                  | Use                                              |
| --------------------- | ------------------------------------------------ |
| `list_roblox_studios` | enumerate connected Studio sessions              |
| `set_active_studio`   | select one exact discovered session              |
| `get_studio_state`    | observe selected-session data-model state        |
| `execute_luau`        | run only Worldwright's fixed bridge in Edit mode |

The discovered `execute_luau` input schema must support a Luau source or code string,
`datamodel_type`, and the Edit value. Missing tools produce `studio.tool_missing`; incompatible
schemas produce `studio.tool_schema_unsupported`. The client never guesses undocumented arguments.

Optional tools are `search_game_tree`, `inspect_instance`, and `screen_capture`. General inspection
may cross-check evidence but is never the canonical snapshot source. Capture is optional for package
transactions and required for the complete live milestone proof. The package does not call tools for
asset search, insertion, mesh/material/procedural generation, image upload or storage, HTTP, or
subagents.

## Session listing, selection, and probing

The public workflow lists sessions, selects an exact ID, and probes the selected session. IDs and
display names are opaque bounded strings.

- Mutating commands always require `--studio-id <exact-id>`, even when one Studio is connected.
- `probe` may list sanitized sessions without selecting a mutation target.
- Read-only `snapshot` and `plan-live` may auto-select only when exactly one Studio exists; explicit
  selection remains preferred.
- Multiple sessions without an exact ID fail with `studio.session_ambiguous`, list sanitized
  candidates, and perform no mutation.
- An unknown ID fails with `studio.session_not_found`.
- Focus, list order, display name, and active status do not authorize mutation.

The fixed probe observes Studio ID, place name, `game.PlaceId`, `game.GameId`, data-model mode,
playtest state, and Edit execution availability.

An exact ID is private local selection state, not shareable run evidence. The live-smoke runner uses
the command-line value for selection and re-selection but omits it from its pre-mutation review and
sanitized summary. Probe output and strict `0.1.0` receipts may contain the ID locally; those raw
outputs stay ignored and private and must not be copied into committed artifacts or pull requests.

## Sandbox restrictions

Probe may report high-level state for a non-sandbox place. Reading managed project contents,
planning against live state, applying a change set, verifying a manifest, or capturing milestone
evidence requires:

```text
game.PlaceId == 0
game.GameId == 0
data model == Edit
playtesting == false
```

A cloud-associated or published place returns `studio.published_place_forbidden`. Play, Run,
simulation, or unavailable Edit execution returns `studio.edit_mode_required`. There is no bypass
flag, environment variable, place allowlist, or implicit approval. See
[Sandbox setup](sandbox-setup.md).

The host probe is not the only gate. Every non-probe fixed bridge program rechecks the place and run
state inside Studio immediately before dispatch, so a place loaded, published, or started between
the probe and an action fails before managed project state is read or mutated.

## Bridge requests, programs, and responses

The bridge supports only these actions:

| Action     | Purpose                                                        |
| ---------- | -------------------------------------------------------------- |
| `probe`    | report fixed sandbox and data-model facts                      |
| `snapshot` | observe bounded project state and ownership boundaries         |
| `create`   | create one validated complete compiler node                    |
| `update`   | replace one complete node under a complete-before precondition |
| `delete`   | delete one exact verified leaf after ownership checks          |

There is no `execute`, `eval`, `run_script`, generic `set_property`, arbitrary `create_instance`,
asset operation, or raw action. Mutation payloads carry only strict managed nodes from existing
compiler contracts. When a create or update resolves a managed parent, the request also carries that
parent's complete canonical node JSON and lowercase SHA-256 exactly as observed by the enclosing
transaction. Workspace-root operations carry no managed-parent state.

The compiler node contract remains unchanged, but the Studio adapter adds the engine-specific
`Instance.Name` boundary: every managed request node, including before, after, and observed parent
state, must contain at most 100 Unicode scalar values. The fixed Luau counts UTF-8 code points again
before mutation and rejects malformed UTF-8, surrogate code points, and longer names.

Every program combines one audited fixed bridge implementation, one schema-validated JSON payload,
and one deterministic long-bracket literal. The encoder chooses the minimum `=` count for which the
closing delimiter is absent. Dynamic strings, class names, and property names remain JSON data; none
are interpolated into executable syntax. `HttpService:JSONDecode` and `JSONEncode` are allowed only
for this local framing.

The bridge returns exactly `WORLDWRIGHT_STUDIO_BRIDGE_V1\n` followed by one strict JSON value and a
final newline. Missing or repeated prefixes, trailing non-whitespace, malformed JSON, duplicate
object keys, unknown fields, unsupported versions, and output beyond the 96 KiB framed limit are
rejected. JSON whitespace and valid runtime-specific scalar encoding need not be byte-identical to
Node serialization; the client normalizes the validated value in Node. Console text is never parsed
as a success.

Snapshot success carries `compactSnapshot`, a deterministic columnar/tuple representation. Its
string and scalar dictionaries are sorted and unique, with names maximally front-coded by Unicode
scalar value and malformed surrogate sequences rejected; managed nodes are sorted by complete entity
ID; and unmanaged roots are sorted by parent ID, class, name, and ordinal. Fixed tuples use
zero-based dictionary and node indexes, with `-1` only in defined parent, source-hash, and shape
absence positions. Primitive tuples encode their verified canonical node properties through number
and enum dictionaries plus a closed bit mask for `Anchored`, collision, query, touch, and shadow
flags. The stored SHA-256 for each sorted node is packed in the same order as exactly 40 canonical
Z85 characters.

Every decoded compact name is independently limited to 100 Unicode scalar values. This applies to
both managed nodes and observed unmanaged-root names; an overlong unmanaged name rejects the whole
snapshot without mutating it.

Studio verifies the source state before producing this compact form. Before snapshot traversal, the
fixed bridge runs three SHA-256 known vectors and fails closed if its runtime implementation
disagrees. It computes SHA-256 over the exact raw state-JSON bytes and requires equality with the
stored state hash, validates the decoded metadata, and verifies the actual hierarchy, attributes,
class, name, and allowlisted engine properties. The host then rejects non-maximal or invalid
front-coding, malformed or noncanonical Z85, noncanonical dictionaries or ordering, incorrect tuple
lengths, unknown class codes or flag bits, invalid sentinels, out-of-range indexes, invalid or
duplicate identifiers, and inconsistent root, parent, hierarchy, or unmanaged-root semantics. After
reconstruction, it requires each decoded stored hash to equal the Node-computed hash of the
corresponding canonical node. Public compiler snapshot normalization and validation run last. The
compact representation is wire-only and does not change the Roblox Scene Snapshot contract or its
canonical hash.

## Managed attributes and metadata

The adapter verifies and writes the existing public ownership attributes:

- `WorldwrightManaged`
- `WorldwrightProjectId`
- `WorldwrightEntityId`
- `WorldwrightEntityKind`
- `WorldwrightCompilerVersion`
- `WorldwrightSourceHash` where required on the root

It may write exactly three internal attributes:

| Attribute                         | Rule                                                       |
| --------------------------------- | ---------------------------------------------------------- |
| `WorldwrightStudioAdapterVersion` | exactly `0.1.0`                                            |
| `WorldwrightStudioStateJson`      | canonical normalized node JSON, subject to a length bound  |
| `WorldwrightStudioStateHash`      | lowercase SHA-256 of canonical node JSON, computed in Node |

Internal metadata is excluded from Manifest nodes and Snapshot node attributes and changes no
compiler wire contract. It cannot contain complete documents, manifests, change sets, prompts,
reference URLs, credentials, machine paths, or reasoning traces.

## Supported classes and properties

The class set exactly matches compiler `0.1.0`: `Folder`, `Model`, `Part`, `WedgePart`, and
`CornerWedgePart`. The bridge uses an explicit class switch; an arbitrary payload string is never
passed directly to `Instance.new`.

For `Folder` and `Model`, verification covers class, name, direct parent, public managed attributes,
and adapter metadata. Primitive verification additionally covers `CFrame`, `Size`, `Anchored`, shape
where applicable, `Material`, `Color`, `Transparency`, `CanCollide`, `CanQuery`, `CanTouch`, and
`CastShadow`. Explicit fixed setters cover this closed set; payload keys are not used as dynamic
property or attribute names.

## Engine-state verification

A live managed Instance is valid only when Studio's SHA-256 of the exact raw stored node-JSON bytes
matches the stored hash, the decoded value and public attributes validate, the adapter version is
current, IDs and project scope agree, actual class/name/direct parent agree, and all actual
allowlisted properties agree. For compact snapshots, the host independently decodes the packed Z85
hash and requires it to match the Node-computed hash of the reconstructed canonical node.

Primitive rotation is reconstructed as:

```lua
CFrame.new(position)
    * CFrame.fromEulerAnglesXYZ(math.rad(x), math.rad(y), math.rad(z))
```

Finite numbers, vectors, colors, sizes, and `CFrame` components that pass through Roblox conversion
use epsilon `0.00001`. Boolean, class, enum, identity, name, and hierarchy comparisons are exact.
Property disagreement is `studio.engine_state_drift`; missing or malformed metadata is
`studio.adapter_metadata_invalid`, and encoded node metadata over 256 KiB is
`studio.adapter_metadata_too_large`. All perform no mutation. Manual edits are not silently adopted.

## Snapshot rules

`readSnapshot` runs one fixed snapshot bridge, performs the required read-only single pass over at
most 65,536 Workspace Instances, bounds selected-project nodes at 2,048, verifies raw metadata
hashes, decoded semantics, and live engine state in Studio, then emits only deterministic compact
transport fields within the 96 KiB framed-output cap. The structural pass detects selected-project
nodes outside their root but records only direct unmanaged boundaries; it does not serialize
protected subtrees. Per-mutation lookup is narrower: it scans only direct Workspace roots to find
the exact selected root, then indexes selected-project managed descendants without crossing an
unmanaged or foreign-project boundary. Node code strictly decodes maximally front-coded names and
node-order Z85 state hashes, reconstructs canonical nodes, requires each stored hash to match its
reconstructed canonical node, then normalizes and validates the snapshot through
`@worldwright/roblox-compiler` and returns a deep-independent value.

An empty selected project has no root, nodes, or unmanaged roots. A non-empty project requires
exactly one `Folder` or `Model` root that is a direct child of `Workspace`, has no managed parent,
belongs to the selected project, and carries the required source hash. Every other node requires a
direct selected-project managed parent. Duplicate IDs, missing parents, cycles, nodes outside the
root, unsupported classes, and malformed hierarchy fail closed.

Foreign-project nodes are excluded. The first foreign-project root nested beneath selected managed
content becomes protected unmanaged content.

## Unmanaged-root identity and protection

An unmanaged root is a direct child of a selected-project managed node that is not managed by that
project. The bridge does not mutate it, write attributes, rename it, clone or serialize its subtree,
or use its display name alone as identity.

The bridge returns a descriptor containing direct managed parent entity ID, child class and name,
canonical structural path, and deterministic duplicate ordinal. The ordinal is counted within the
same-class-and-name sibling group, and the path is `<parent-id>/<class>/<name>/<duplicate-ordinal>`.
Unrelated sibling insertion or ordering therefore does not perturb existing markers. Node code
hashes the canonical descriptor into a bounded `unmanaged-<hex-fragment>` snapshot ID. This is
structural, observation-local identity, not a permanent Roblox object ID. `Instance.UniqueId` is
unavailable to ordinary Edit-mode script access and `GetDebugId` requires Plugin Security. Rename,
reparent, class, path, or duplicate-group membership can change the snapshot hash.

Delete and reparent fail when a target lineage contains an unmanaged or foreign-project root.

## Mutation behavior

### Create

Create validates a complete node, finds exact project and parent identities, requires the ID to be
absent, and verifies the resolved managed parent against the transaction-observed canonical parent
node, state JSON, state hash, and actual engine properties. Self-consistent but concurrently changed
parent metadata is not accepted. It then creates one allowlisted Instance detached, applies explicit
properties and ownership attributes, writes metadata, parents last, and rereads the exact result. A
pre-parent failure destroys the detached Instance; a post-parent failure may remove only the
just-created matching managed node with no children. Cleanup is protected and reindexes to verify
absence. A proven clean failure is `studio.create_failed`; an exception or uncertain remainder is
`studio.create_cleanup_failed`.

### Update

Update requires matching before/after IDs and classes, exact project scope, and live state equal to
the complete `before` node. The destination parent must equal its exact transaction-observed state;
reparenting is also blocked by protected descendants. Explicit setters run in a protected call,
metadata is written last, and the complete `after` state is verified. On failure, the same bridge
call verifies the original parent against its transaction-observed state before attempting to
restore and verify the complete before state. A partial write is never reported as success.

### Delete

Delete locates the project/entity target, verifies complete `before` state, requires no selected
managed children and no unmanaged or foreign-project root in the subtree, destroys only that exact
Instance, and confirms the ID is absent. It never performs arbitrary recursive destruction.

## Transaction behavior

The adapter implements only `readSnapshot`, `createNode`, `updateNode`, and `deleteNode`.
`applyRobloxChangeSet` remains the transaction authority. It validates and freshly observes the
base, rejects stale state before mutation, purely simulates the transition, returns `noop` without a
mutation when appropriate, applies ordered operations, and verifies an independent final snapshot.

Within that authorized transaction, every executor snapshot read replaces an expected-node cache
with the newly verified managed nodes. Successful create, update, and delete calls advance the cache
for later operations in the same transition. Create, update, and operation-local restoration derive
their managed-parent preconditions from this cache; this does not widen or modify the compiler
adapter and change-set contracts.

After apply or verification failure, compensation proceeds only from an observed state within the
attempted-operation envelope. Restoration is reported only when the complete initial snapshot hash
is recovered. More than 512 Studio operations are rejected before execution. Version `0.1.0` uses
one fixed `execute_luau` call per mutation; batching is future work.

  `studio.tool_schema_unsupported`, `studio.tool_call_failed`, and `studio.tool_timeout`;

  `studio.session_ambiguous`;
- `studio.edit_mode_required`, `studio.published_place_forbidden`, and `studio.project_mismatch`;
- `studio.node_limit_exceeded`, `studio.operation_limit_exceeded`, `studio.payload_too_large`,
  `studio.identity_invalid`, `studio.root_invalid`, and `studio.hierarchy_invalid`;
- `studio.class_unsupported`, `studio.property_invalid`, `studio.adapter_metadata_invalid`,
  `studio.adapter_metadata_too_large`, and `studio.engine_state_drift`;
- `studio.unmanaged_content_protected`, `studio.foreign_project_protected`, `studio.create_failed`,
  `studio.create_cleanup_failed`, `studio.update_failed`, `studio.update_restore_failed`, and

  `studio.capture_invalid`, and `studio.receipt_invalid`; and

Diagnostics exclude raw Luau, MCP stack traces, console output, stderr, paths, usernames,
credentials, and environment dumps.

## Studio Apply Receipts

The receipt is a strict `applied`, `noop`, or `failed` union. It records schema and adapter
versions, sanitized sandbox identity, project and target, change-set/base/desired/expected hashes,
planned and attempted operations, status, deterministic diagnostics, failure stage, rollback outcome
and observed hashes where applicable, and optional viewport media type, lowercase SHA-256, and byte
length.

It excludes timestamps, paths, raw images, complete snapshots or change sets, raw MCP responses,
console output, credentials, usernames, and environment values. Its normalized hash is
deterministic. A receipt is observed-result evidence, not authorization, a signature, or visual
quality proof.

The strict `0.1.0` receipt includes sanitized sandbox identity, including the Studio ID. It is
therefore private, ignored local evidence. Shareable live-smoke review material is the pre-mutation
review and sanitized `summary.json`, both of which omit the exact Studio ID; raw receipts and probe
output must not be committed or pasted into a pull request.

## Viewport capture

Capture requires discovered `screen_capture`, exact lowercase `image/jpeg` output, and an explicit
path beneath the repository-owned `.worldwright/live-milestone-3/`; URLs, other destinations, and
junction escapes are rejected before connection. Receipt outputs use the same anchored boundary. The
CLI reserves an output exclusively before capture or mutation, writes validated bytes once, and
reports only media type, lowercase SHA-256, and byte length. The receipt never stores the path or
image bytes. Live evidence remains untracked.

## CLI

The root command is `pnpm studio-mcp`:

```sh
pnpm studio-mcp probe
pnpm studio-mcp snapshot --studio-id <id> --project-id <project-id> --output <snapshot-file>
pnpm studio-mcp plan-live --studio-id <id> --manifest <manifest-file> --output <change-set-file>
pnpm studio-mcp apply --studio-id <id> --change-set <change-set-file> --confirm <full-change-set-sha256> --receipt-output <receipt-file>
pnpm studio-mcp verify --studio-id <id> --manifest <manifest-file>
pnpm studio-mcp capture --studio-id <id> --output .worldwright/live-milestone-3/<image-file.jpg>
```

Commands support `--json` where practical. Output paths must be explicit and cannot overwrite an
input. `apply` requires the full lowercase normalized change-set SHA-256. It rejects prefixes,
`yes`, `y`, `--yes`, `--force`, environment confirmation, or the manifest hash. Human output shows
the sandbox, project, operation counts, transition hashes, and required hash before mutation.
`plan-live` and `apply` remain separate. No option accepts raw Luau, an MCP executable or URL, or a
published-place bypass.

| Exit code | Meaning                                                                      |
| --------: | ---------------------------------------------------------------------------- |
|         0 | success, including verified no-op                                            |
|         1 | domain failure, stale state, conflict, or failed transaction                 |
|         2 | usage, I/O, MCP connection, unavailable Studio, or incompatible tool failure |

## Live smoke setup

Run live acceptance separately from ordinary CI:

```sh
pnpm studio:live-smoke -- --review
pnpm studio:live-smoke -- --studio-id <exact-studio-id> --confirm <full-reviewed-live-sequence-sha256>
```

It must use the unsaved sandbox in [Sandbox setup](sandbox-setup.md). The checked-in Cliffwatch
manifest is used to prove create, no-op, one harmless name update and repair, controlled
post-mutation failure with compensation, JPEG viewport capture, receipt validation, and final
restoration to the canonical manifest. `--review` performs no connection or mutation and prints the
strict sequence envelope plus its complete lowercase authorization hash. The real run accepts only
that full hash; the envelope pins the create, canonical no-op, one-node update, inverse repair,
controlled-fault, capture-media, and final-no-op sequence. Prefixes, symbolic approval, different
full hashes, and fixture drift fail before connection. After connection, the command matches the
actual empty or canonical initial state to its exact reviewed transition and prints a JSON-escaped
pre-mutation review before its first mutation. That review names the unsaved place and reviewed
transition details but deliberately omits the exact Studio ID. Every other non-empty state fails
closed.

Evidence is local and untracked. All five fixed outputs are exclusively reserved before connection;
an incomplete run closes and removes every reservation. The sanitized `summary.json` excludes Studio
ID, place name, machine paths, raw messages, and image bytes, records strict receipt validation, and
states whether the optional live ownership-boundary check ran. The three raw receipts retain their
contractual sanitized sandbox identity and therefore remain private local evidence, not shareable
review artifacts. Existing evidence files are never overwritten; move or delete old untracked
evidence before intentionally rerunning. A real Studio run is required before a live-success claim;
fake-client tests do not substitute. `pnpm check` and CI never require Studio.

The regular smoke cannot create an unmanaged or foreign-project boundary: doing so would violate the
five-action bridge. Its ownership protection is therefore offline fake-MCP proof unless the separate
manual Explorer procedure in the sandbox guide is completed and recorded. Do not describe offline
coverage as a live ownership-boundary result.

## Security and limitations

Adapter `0.1.0` guarantees local stdio only; exact Studio selection; unsaved Edit-only places; fixed
bridge actions; inert JSON payloads; closed classes, properties, attributes, and operations; no raw
Luau, scripts, dynamic evaluation, assets, or network; actual engine verification; no unmanaged or
foreign-project mutation; stale rejection; complete before checks; exact result verification;
verified compensation; bounded resources; sanitized errors; and untracked evidence.

It adds no plugin, Forge, ChangeHistoryService, AI provider, HTTP service, database, authentication,
telemetry, analytics, or deployment. It does not implement Play/Run, gameplay or traversal testing,
visual critique, The Critic, Atlas, manual drift adoption, permanent unmanaged identity,
high-throughput batching, published-place authorization, or engine-level transaction isolation.

Creators should connect only trusted MCP clients. Creator edits during a transaction can make
compensation inadmissible; Worldwright then fails without overwriting the uncertain change.

## Versioning

Adapter, bridge, and receipt versions are explicit and independent of WorldSpec, Architecture
Planner, and compiler versions. Unsupported versions are rejected. Patch releases may correct
behavior without changing accepted meaning; compatible additions require a minor contract version;
removed, renamed, narrowed, or reinterpreted fields, actions, framing, hash inputs, attributes,
selection rules, ownership behavior, or sandbox authorization require a major version.

WorldSpec `0.1.0`, all four Roblox compiler `0.1.0` wire contracts, and Architecture Planner `0.1.0`
remain unchanged.
