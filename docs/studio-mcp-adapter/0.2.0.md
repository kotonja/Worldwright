# Studio MCP Adapter 0.2

## Scope

`@worldwright/studio-mcp-adapter` package `0.2.0` adds deterministic chunked mutation and
reconnectable observation after an uncertain Studio MCP outcome. It remains a private,
local-stdio-only adapter for one exact unsaved Roblox Studio session in stopped Edit mode.

Adapter `0.2.0` does not change WorldSpec, Architecture Planner, Roblox directive, Manifest, Scene
Snapshot, Change Set, Studio bridge request/response, or Studio Apply Receipt `0.1.0` wire meaning.
It introduces separate batch and report contracts instead of adding an action to the published
closed single-action bridge union. It also introduces a separate sandbox-lease protocol because an
exact Studio ID selects a Studio process but is not sufficient identity for the unsaved DataModel
currently loaded inside it.

## Compatibility with Milestone 3 nodes

The npm package version and persisted node-metadata version are deliberately separate:

| Meaning                                           | Constant                         | Value   |
| ------------------------------------------------- | -------------------------------- | ------- |
| Installed adapter package                         | `STUDIO_MCP_PACKAGE_VERSION`     | `0.2.0` |
| Persisted `WorldwrightStudioAdapterVersion` value | `STUDIO_STORED_METADATA_VERSION` | `0.1.0` |
| Backward-compatible persisted-version alias       | `STUDIO_MCP_ADAPTER_VERSION`     | `0.1.0` |

Instances created by adapter `0.1.0` remain readable. Their existing canonical node JSON and hashes
retain the same meaning, and unchanged nodes do not require a 400-node metadata migration merely
because the host package version advanced.

The original fixed `probe`, `snapshot`, `create`, `update`, and `delete` actions and their
`WORLDWRIGHT_STUDIO_BRIDGE_V1` framing remain version `0.1.0`. Sequential compiler adapters remain
supported.

## New contracts

All new domain values are strict JSON-only closed objects, are normalized deterministically, reject
unknown fields and unsupported versions, and have generated draft 2020-12 JSON Schemas:

| Contract                      | Version | Schema ID                                             |
| ----------------------------- | ------- | ----------------------------------------------------- |
| Studio Batch Request          | `0.1.0` | `urn:worldwright:studio-batch-request:0.1.0`          |
| Studio Batch Response         | `0.1.0` | `urn:worldwright:studio-batch-response:0.1.0`         |
| Studio Sandbox Lease Record   | `0.1.0` | `urn:worldwright:studio-sandbox-lease-record:0.1.0`   |
| Studio Sandbox Lease Request  | `0.1.0` | `urn:worldwright:studio-sandbox-lease-request:0.1.0`  |
| Studio Sandbox Lease Response | `0.1.0` | `urn:worldwright:studio-sandbox-lease-response:0.1.0` |
| Studio Progress Report        | `0.1.0` | `urn:worldwright:studio-progress-report:0.1.0`        |

The Studio Apply Receipt remains the separate closed `urn:worldwright:studio-apply-receipt:0.1.0`
contract. Transport fields are not inserted into it.

## Batch request contract

A request has this top-level shape:

```json
{
  "protocolVersion": "0.1.0",
  "action": "apply_chunk",
  "projectId": "project-id",
  "changeSetHash": "<64 lowercase hex>",
  "sandboxLeaseId": "<private 64 lowercase hex>",
  "chunkId": "<64 lowercase hex>",
  "chunkIndex": 0,
  "operations": []
}
```

`operations` contains one to 32 strict values and retains the Change Set's canonical order.

### Create

A create contains:

- `type: "create"` and exact `operationId`;
- the complete desired managed `node`;
- its canonical `stateJson` and lowercase SHA-256 `stateHash`; and
- optional complete `parentState` when the node has a managed parent.

### Update

An update contains:

- `type: "update"` and exact `operationId`;
- complete `before` and `after` nodes with identical ID and class;
- canonical before/after JSON and hashes; and
- optional exact before- and after-parent states where required.

### Delete

A delete contains:

- `type: "delete"` and exact `operationId`;
- the complete `before` node; and
- canonical before JSON and hash.

Every operation must pass the existing single-action bridge validation. Operation IDs and target IDs
are unique inside a chunk, and create/update/delete phase order cannot regress. The contract has no
arbitrary class, property or attribute map, Luau, URL, credential, asset, manifest, WorldSpec, or
complete Change Set field.

`chunkId` is recomputed from the canonical project, Change Set hash, chunk index, ordered operation
IDs, and complete operation contents. The private sandbox lease ID is deliberately excluded from
chunk identity, reports, receipts, and shareable hashes. A caller-provided mismatched ID fails
validation.

## Transaction-scoped sandbox lease

For every nonempty authorized transaction, the adapter stores one strict canonical JSON record in
exactly `Workspace:GetAttribute("WorldwrightStudioSandboxLeaseJson")`. The closed record contains
only schema version `0.1.0`, a cryptographically random 32-byte Node-generated lease ID, project ID,
and complete Change Set hash. It is private transport evidence, not authentication, a signature,
creator approval, permanent Roblox identity, snapshot evidence, or mutation authorization. The
complete normalized Change Set hash remains authorization.

The separate fixed `read_lease`, `claim_lease`, and `bound_snapshot` actions use
`WORLDWRIGHT_STUDIO_SANDBOX_LEASE_V1\n` framing. Claim compares the exact previously read canonical
record or explicit absence, writes the new record with one `SetAttribute` call, and verifies exact
readback. Malformed existing data fails closed without overwrite. The record remains after the
transaction; the next nonempty transaction rotates it through the same compare-and-set protocol. A
verified no-op returns before preparation and performs zero lease claims and zero mutation calls.

Worldwright may read or write only that one Workspace attribute through the fixed lease protocol in
an unsaved stopped sandbox. It does not enumerate, copy, clear, or expose other Workspace
attributes, copy the lease into managed-node metadata, or use a published-place path.

## Chunk bounds

Production constants are:

```text
STUDIO_MCP_MAX_BATCH_OPERATIONS = 32
STUDIO_MCP_MAX_BATCH_PAYLOAD_BYTES = 3 * 1024 * 1024
STUDIO_MCP_BATCH_TOOL_TIMEOUT_MS = 45_000
STUDIO_MCP_MAX_RECONNECTS_PER_TRANSACTION = 2
```

The canonical request must also fit the existing 4 MiB outer payload bound. A planner closes a chunk
before adding an operation that would cross either the operation or byte limit. It never emits an
empty chunk or splits one operation. An individually oversized operation fails before a Studio tool
call.

The overall transaction limit remains 512 operations. The checked-in 400-create Cliffwatch
transition deterministically plans 13 chunks, within the required maximum of 16 mutation calls for a
successful forward create.

## Fixed batch program

The adapter constructs one fixed `apply_chunk` Luau program from audited source and one inert,
schema-validated JSON literal. The builder and source are not exposed as raw execution APIs.

Inside Studio, the program:

1. rechecks unsaved `PlaceId == 0` and `GameId == 0`;
2. rejects a running data model;
3. validates the closed payload and operation bound;
4. verifies the complete current Workspace lease record against the request's lease ID, project ID,
   and Change Set hash, returning a strict zero-operation identity failure on any mismatch;
5. builds the exact selected-project managed root and index once for the chunk;
6. processes operations in exact order while updating that in-memory root and index after each
   acknowledged operation;
7. invokes the existing audited create, update, or delete behavior for each operation;
8. verifies complete before/after state, managed parent, ownership, class, metadata, and actual
   allowlisted engine properties through the shared helpers;
9. records the exact completed operation prefix;
10. stops at the first failure and attempts only the existing operation-local cleanup or restoration
    for that failing operation;
11. after local completion, rebuilds the selected-project index once and requires its root and
    complete entity-to-Instance mapping to equal the in-memory index; and
12. verifies every requested create or update in its complete after state and every requested delete
    as absent before returning success.

It does not perform transaction-level rollback. The compiler transaction engine decides whether
freshly observed state is compensable. A failed authoritative end-of-chunk rebuild or complete-state
check is not reported as a successful acknowledgment and does not claim local restoration.

## Batch response contract

Batch responses use exact `WORLDWRIGHT_STUDIO_BATCH_V1\n` framing, one JSON object, and one final
newline. The complete framed value is capped at 96 KiB.

Success includes:

- matching protocol, action, Change Set hash, chunk ID, and chunk index;
- `ok: true`;
- `operationsAttempted` and `operationsApplied`; and
- `completedOperationIds`.

A success must attempt and apply every requested operation. The completed IDs must be the complete
exact requested prefix.

Failure includes the same identity and count fields plus:

- `ok: false`;
- optional `failedOperationId` for the next operation after the completed prefix;
- `localRestoreSucceeded`; and
- one strict bridge diagnostic.

For either variant, `operationsApplied <= operationsAttempted <= request.operations.length`, and the
completed-ID count equals `operationsApplied`. A malformed, duplicate-key, unknown-field,
version-mismatched, identity-mismatched, incomplete, oversized, or ambiguously framed response is an
uncertain transport outcome rather than a trustworthy partial acknowledgment.

`localRestoreSucceeded: true` is accepted only for a pre-operation gate failure or a failed
operation whose diagnostic is compatible with proven operation-local restoration. Cleanup, restore,
delete, response-integrity, and post-chunk verification failures cannot claim restoration; an
impossible combination makes the complete response uncertain and poisons the lane.

## Transaction behavior

`applyRobloxChangeSetBatched` and the original sequential `applyRobloxChangeSet` share the
compiler's transaction engine. Both require:

- valid normalized inputs;
- a fresh exact base snapshot;
- `baseSnapshotHash` equality before mutation;
- complete pure simulation;
- canonical operation order;
- an independent complete final snapshot; and
- exact `resultSnapshotHash` equality before success.

The Studio adapter prepares parent state from the transaction's current expected node map. Preparing
operations advances that map in canonical order, allowing a child to depend on a parent created or
updated earlier in the same chunk or transaction. Runtime state advances only for a strict
acknowledged prefix, and a fresh snapshot replaces the map completely.

A batch response is never accepted as the final snapshot. Public `ApplyResult.operationsAttempted`
counts node operations, while the separate transport report counts chunks and MCP calls.

The shared compiler engine calls the optional Studio preparation hook only after validation, an
exact fresh base, complete pure simulation, and the early no-op return. After the compare-and-set
claim, it requires a second complete `bound_snapshot` with the exact original base hash before batch
planning or the first node mutation. Every later forward, final-verification, failure-observation,
compensation, and post-compensation call remains bound to that same record. Preparation or changed
base failure reports zero attempted node operations and does not compensate.

## Certain and uncertain failure

A valid failure response with an exact prefix and proven operation-local restoration is a certain
failure. The compiler observes state and considers compensation within that exact attempted
envelope.

These cases are uncertain:

- timeout or rejected `execute_luau`;
- closed or terminated client process;
- response lost after Studio execution;
- missing, malformed, mismatched, incomplete, duplicate-key, trailing, or oversized response; or
- a response that does not prove operation-local restoration.

Every operation submitted in an unresolved chunk is conservatively counted as attempted. The adapter
never retransmits that chunk.

## Exact-session reconnection

After an uncertain mutation the current client is poisoned and its owned process tree is closed. No
later call is made through it. The next required snapshot observation may establish a new default
local-stdio client.

Before the replacement is trusted, Worldwright rediscovers required tools, lists Studio sessions,
selects the exact original Studio ID, confirms it is active, and re-probes the sandbox. The place
must still have zero PlaceId and GameId and be stopped in Edit mode. It then verifies the original
transaction lease and obtains the complete selected-project snapshot together in one fixed
`bound_snapshot` call. The candidate client is not accepted before that observation succeeds.

A different sole session, same display name, focus, active flag, list order, published place, or
running data model is rejected. The same Studio ID with an absent, malformed, or different lease,
project, or Change Set is a different DataModel for recovery: it yields
`studio.sandbox_identity_mismatch`, performs zero compensation, and cannot be reported restored.
Reconnection is limited to two attempts per transaction. The normal allocation is one after
uncertain forward application and one after uncertain compensation; there is no unbounded loop.

A rejected replacement is closed before another attempt. If its bounded close fails or times out,
that candidate process tree is unproven and the lease permanently refuses any further replacement or
automatic recovery in the transaction.

## Progress classification

The compiler's pure `classifyRobloxChangeSetProgress` validates a reviewed base snapshot, a fresh
observed snapshot, and the Change Set. Its successful classifications are:

| Classification | Meaning                                                                          |
| -------------- | -------------------------------------------------------------------------------- |
| `base`         | Observation equals the exact base at prefix zero.                                |
| `prefix`       | Observation equals one exact nonzero proper prefix in canonical operation order. |
| `complete`     | Observation equals the state after all operations.                               |

An unsafe result contains deterministic `progress.*` diagnostics. Project or target mismatch, stale
base hash, changed unmanaged roots, malformed hierarchy, invalid operation preconditions, arbitrary
subsets, skipped or reordered effects, third-state targets, and unrelated managed additions, edits,
or deletions are unsafe.

The classifier builds node maps and one initial mismatch count, then updates only the targeted ID as
each expected operation is applied. It does not rerun complete-prefix simulation or serialize a
complete snapshot in every loop.

## Conservative recovery policy

Batch protocol `0.1.0` never automatically resumes uncertain forward work and never blindly retries
the uncertain chunk.

- Observed `base`: fail the original transaction without compensation.
- Observed exact `prefix`: if it is within the conservative attempted envelope, plan compensation to
  the original base and verify the full base hash.
- Observed `complete`: report the original forward transaction failed, compensate the full prefix to
  base, and verify the full base hash.
- Observed `unsafe`: perform zero compensation mutations.

A pure snapshot transition must first prove the inverse is safe. The compiler then executes the
exact inverse of the observed authorized prefix in reverse order, so an uncertain partial
compensation remains a shorter exact forward prefix that can be independently classified after the
second bounded reconnect.

Replanning after uncertain compensation requires a strictly shorter observed forward prefix. If a
fresh snapshot shows zero compensating progress, Worldwright reports `failed-unrestored` and does
not retransmit the same uncertain chunk.

A creator may explicitly rerun the complete confirmed Change Set only after the base has been
verified. Restoration is successful only when a fresh complete snapshot equals the original base
hash.

## Progress CLI

Read-only classification uses:

```sh
pnpm studio-mcp progress \
  --studio-id <exact-studio-id> \
  --sandbox-lease-id <64-lowercase-hex> \
  --base-snapshot <reviewed-base-snapshot.json> \
  --change-set <reviewed-change-set.json>
```

Add `--json` for the strict report envelope and deterministic report hash. The explicit base file is
required because a Change Set contains the base hash, not the complete base snapshot needed to prove
an exact prefix.

The command proves exact Studio selection and the sandbox gate, then verifies the supplied private
lease and reads one complete project snapshot together through `bound_snapshot`. It performs no
claim or mutation and never prints or reports the lease. A missing or mismatched lease cannot return
a base, prefix, or complete classification.

| Exit code | Meaning                                               |
| --------: | ----------------------------------------------------- |
|         0 | Valid `base`, `prefix`, or `complete` classification. |
|         1 | Unsafe/non-prefix state or invalid domain input.      |
|         2 | Usage, I/O, MCP, Studio, or tool failure.             |

There is no `--resume`, `--force`, `--clear`, `--delete-managed`, `--adopt`, or `--repair-anyway`
option, and adapter `0.2.0` exposes no automatic recovery command.

## Apply CLI

Mutation authorization remains:

```sh
pnpm studio-mcp apply \
  --studio-id <exact-studio-id> \
  --change-set <reviewed-change-set.json> \
  --confirm <complete-normalized-change-set-sha256>
```

The preview adds transport mode, planned chunk count, and maximum operations per chunk. It does not
print raw operation payloads. Human and JSON completion output includes transaction status,
operation count, chunk, sandbox-lease-claim, and mutation-call counts, reconnect attempts,
compensation restoration, receipt hash, and transport-report hash.

Confirmation is never a chunk hash. Prefixes, symbolic approval, manifest hashes, and environment
confirmation remain invalid.

## Studio Transport Report

The strict report has `schemaVersion: "0.1.0"`, `mode: "chunked"`, the complete Change Set hash, and
these deterministic counts:

- operations planned, attempted, and applied before forward failure;
- forward chunks planned, attempted, and completed;
- sandbox lease claim calls;
- total mutation `execute_luau` calls;
- uncertain transport events;
- reconnect attempts and successes;
- compensation operations attempted and applied;
- compensation chunks attempted and completed; and
- final outcome: `applied`, `noop`, `failed-restored`, `failed-unsafe`, or `failed-unrestored`.

Counts are validated for internal consistency. A no-op records zero sandbox lease claim calls. A
transaction that reaches lease-bound observation, forward mutation, reconnect, or compensation
records exactly one claim call; reconnect and compensation reuse that transaction lease rather than
claiming another. `mutationExecuteCalls` counts forward and compensation batch programs only, not
the separate claim call. Normally it equals attempted forward plus compensation chunks. A
`failed-unrestored` compensation-time lease rejection may add exactly one execute call without
incrementing compensation counters because the fixed batch call reached the lease guard but
performed no operation. The report exposes the count but no lease ID, lease record, Workspace lease
attribute contents, timestamp, Studio ID, place name, path, raw MCP message, raw Luau, image bytes,
environment value, username, or credential. Its canonical hash is deterministic.

The strict counter maxima account for the one observation-gated compensation replan: at most 544
compensation-operation attempts or reported applications, 544 compensation-chunk attempts, and 1,056
total mutation calls. Definite inverse progress is subtracted before the replan, so only the
unresolved batch of at most 32 operations remains. These evidence bounds do not authorize blind
retry or enlarge the 512-operation transaction. `applied` and `noop` require no uncertainty,
reconnect, or compensation evidence; `failed-unsafe` requires zero compensation; and reconnect
attempts cannot exceed recorded uncertainty events. A late unsafe observation after compensation
started is represented as `failed-unrestored`, not `failed-unsafe`. An `applied` report also
requires between `ceil(operationsPlanned / 32)` and `operationsPlanned` nonempty forward chunks,
with matching attempted, completed, and mutation-call counts.

The private Studio Apply Receipt continues to record transaction status and exact snapshot hashes.
CLI JSON or live evidence may carry a receipt and transport report beside each other, but neither
contract is widened.

## Live smoke setup

Ordinary `pnpm check` and GitHub Actions remain Studio-independent. Real acceptance is a separate,
explicitly reviewed sequence:

```sh
pnpm studio:batch-live-smoke -- --review
pnpm studio:batch-live-smoke -- \
  --studio-id <exact-studio-id> \
  --confirm <full-reviewed-sequence-sha256>
```

Use a fresh new unsaved local baseplate with Studio MCP enabled, zero PlaceId and GameId, and
stopped Edit mode. Never select a separately open published place.

The sequence claims one lease, applies the canonical 400-create Cliffwatch transition in exactly 13
chunks and 13 batch mutation calls, verifies the exact final hash, proves a zero-mutation and
zero-claim no-op, applies and reverses one harmless name update, and injects one testing-only lost
acknowledgment after Studio completed a reviewed harmless update. The latter must poison the old
connection, reconnect to the exact ID, re-probe the sandbox, obtain the lease-bound snapshot in the
same identity-verification call, classify the exact prefix, compensate under the original lease to
the canonical base, and verify the restored hash.

Evidence stays ignored under `.worldwright/live-milestone-4/`. A shareable summary may contain only
zero PlaceId/GameId, operation/chunk/call/reconnect counts, reviewed hashes, compensation result,
receipt/report hashes, and viewport hash and byte length. Raw receipts containing private Studio
identity, paths, logs, Luau, stderr, image bytes, and machine data must not be committed or pasted
into review. No lease ID, record, or prior Workspace attribute value enters shareable evidence. A
viewport capture is operational evidence, not a visual-quality claim.

See the [recovery runbook](recovery.md) for failure handling.

## Security guarantees

Adapter `0.2.0` retains:

- default local stdio MCP v1 only;
- exact Studio selection, exact-ID reconnect, and same-call lease-bound DataModel observation;
- one fixed, private, compare-and-set Workspace sandbox lease for nonempty transactions;
- unsaved stopped-sandbox enforcement before every managed read or mutation lane;
- fixed audited Luau with one inert validated JSON literal;
- closed class, property, attribute, and action sets;
- actual engine-state and complete parent-state verification;
- unmanaged and foreign-project protection;
- pure compiler preflight, stale rejection, and complete final verification;
- observation-gated compensation and exact restored-hash proof;
- bounded operations, payloads, results, tool timeouts, scans, nodes, reconnects, and process close;
  and
- sanitized deterministic diagnostics and reports.

The root API exposes neither raw `execute_luau` nor arbitrary MCP tool invocation. The batch program
cannot create scripts, load assets, access network or data stores, call `ChangeHistoryService`, or
mutate a published place.

## Limitations and future work

The adapter still provides no engine-level transaction isolation. Creator or competing-client edits
during a transaction may make state unsafe, in which case Worldwright performs no compensation.
Recovery also requires the exact original Studio session to remain available and eligible.

Automatic forward resume is intentionally absent. A later version may define separately reviewed
resume semantics without reinterpreting protocol `0.1.0`.

Milestone 4 does not implement Play or Run mode, character navigation, console collection, gameplay
assertions, collision or visual evaluation, The Critic, Atlas, Forge, a Studio plugin,
ChangeHistoryService, AI calls, assets, databases, authentication, telemetry, deployment, or
published-place authorization.
