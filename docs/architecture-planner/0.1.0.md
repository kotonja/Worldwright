# Architecture Planner 0.1 reference

## Contract summary

`@worldwright/architecture-planner` turns a supported WorldSpec `0.1.0` architectural program into
one deterministic orthogonal Architecture Plan, can emit a compiler-ready derived WorldSpec, and can
verify that document with the Roblox compiler. All steps are offline data transformations.

| Contract                            | Version | JSON Schema ID                                              |
| ----------------------------------- | ------- | ----------------------------------------------------------- |
| Architecture entity directive       | `0.1.0` | `urn:worldwright:architecture-entity-directive:0.1.0`       |
| Architecture relationship directive | `0.1.0` | `urn:worldwright:architecture-relationship-directive:0.1.0` |
| Architecture Plan                   | `0.1.0` | `urn:worldwright:architecture-plan:0.1.0`                   |

The planner implementation version is also `0.1.0`. Directive, plan, and implementation versions are
explicit even while equal; consumers must not assume they always advance together.

TypeBox source schemas are canonical. Generated JSON Schemas use draft 2020-12. Every defined
contract object is closed with `additionalProperties: false`. Public validators accept `unknown`,
reject non-JSON in-memory values, and return stable diagnostics instead of throwing for expected
invalid input.

## Directive locations

The entity directive key is:

```text
entity.attributes["worldwright.architecture"]
```

The relationship directive key is:

```text
relationship.attributes["worldwright.architecture"]
```

The surrounding WorldSpec attribute maps remain open JSON extension points. Each value beneath the
architecture key is a strict, versioned contract. Unknown directive fields fail.

The interior envelope must remain positive after subtracting exterior wall thickness. Across the
perpendicular dimension, the corridor plus both corridor walls must leave at least two grid cells,
one for each room band. All horizontal planning dimensions must align to the grid.

## Floor directive

Mode `floor` is required on every direct `floor` child of the planned structure.

| Field           | Type                      | Meaning and requirements                                                                                                |
| --------------- | ------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `schemaVersion` | literal `0.1.0`           | Entity directive wire version.                                                                                          |
| `mode`          | literal `floor`           | Discriminator.                                                                                                          |
| `level`         | non-negative safe integer | Zero-based level. Levels are unique, contiguous, and limited to 0â€“2.                                                    |
| `clearHeight`   | positive finite number    | Clear wall height; doors must fit, required windows must fit above their sill, and height plus slab must fit the story. |

`defaultClearHeight` is recorded semantic metadata and is not substituted for an authored floor
directive. Every actual floor must fit the default door height. The complete default window must fit
only when that floor has a hard minimum-window request. If only preferred windows are too tall, the
floor remains feasible, emits no invalid windows, and reports `architecture.preference_unsatisfied`.

Finished-floor elevation is `building.origin.y + level * floorToFloorHeight`. Basements, negative
levels, split levels, and mezzanines are unsupported.

## Room directive

Mode `room` is required on every direct `room` child of a planned floor.

| Field                | Type or values                        | Meaning and requirements                                         |
| -------------------- | ------------------------------------- | ---------------------------------------------------------------- |
| `schemaVersion`      | literal `0.1.0`                       | Entity directive wire version.                                   |
| `mode`               | literal `room`                        | Discriminator.                                                   |
| `minimumArea`        | positive finite number                | Hard minimum clear area in square studs.                         |
| `preferredArea`      | positive finite number                | Soft target clear area.                                          |
| `maximumArea`        | positive finite number                | Hard maximum clear area.                                         |
| `minimumSpan`        | positive grid-aligned number          | Hard minimum for either clear rectangle dimension.               |
| `maximumAspectRatio` | finite number at least 1              | Hard maximum of longer clear side divided by shorter clear side. |
| `zone`               | `public`, `private`, or `service`     | Used by deterministic entrance/rear ordering penalties.          |
| `isEntrance`         | boolean                               | Exactly one room is true, and it must be on level zero.          |
| `doorWidth`          | optional positive grid-aligned number | Overrides the building door width for this room's corridor door. |
| `windows.minimum`    | integer from 0 through 64             | Hard minimum exterior windows.                                   |
| `windows.preferred`  | integer from 0 through 64             | Soft preferred windows; must be at least `minimum`.              |

Area ordering is `minimumArea <= preferredArea <= maximumArea`. A room occupies one axis-aligned
clear rectangle, touches its band exterior, touches the corridor, and receives exactly one corridor
door. Its door plus two end clearances must fit the corridor wall.

## Stair directive

Mode `stair` is required on exactly one direct `route` child of a multi-floor building and is not
required for a single-floor building.

| Field                | Type or values                    | Meaning and requirements                                        |
| -------------------- | --------------------------------- | --------------------------------------------------------------- |
| `schemaVersion`      | literal `0.1.0`                   | Entity directive wire version.                                  |
| `mode`               | literal `stair`                   | Discriminator.                                                  |
| `floorIds`           | unique identifier array           | Exactly all planned floor IDs in ascending level order.         |
| `coreWidth`          | positive grid-aligned number      | Perpendicular clear stair-core dimension.                       |
| `coreLength`         | positive grid-aligned number      | Along-corridor stair-core reservation.                          |
| `preferredSide`      | `auto`, `negative`, or `positive` | Selects or candidates the room-band side.                       |
| `position`           | literal `rear`                    | The only v0.1 position, opposite `entranceEnd`.                 |
| `maximumRiserHeight` | positive finite number            | Hard maximum used to derive step count.                         |
| `minimumTreadDepth`  | positive finite number            | Hard minimum after dividing available run length by step count. |

The same core rectangle is used on every floor and must fit a straight run for every adjacent level
pair. The core sits behind a floor-level access lane whose clear depth is exactly
`defaultDoorWidth`; the stair hall still touches the corridor, and its centered corridor opening
enters that lane. The lane provides a clear side route to the first or final tread. The derived step
count is capped at 256 before step geometry can be expanded.

## Relationship directive

Mode `adjacency` is accepted only on an undirected room-to-room `adjacent_to` WorldSpec
relationship.

| Field           | Type or values                      | Meaning                                                            |
| --------------- | ----------------------------------- | ------------------------------------------------------------------ |
| `schemaVersion` | literal `0.1.0`                     | Relationship directive wire version.                               |
| `mode`          | literal `adjacency`                 | Discriminator.                                                     |
| `requirement`   | `required`, `preferred`, or `avoid` | Hard requirement, soft preference, or hard divider avoidance.      |
| `connection`    | `door`, `near`, or `none`           | Requested realization or distance-only scoring.                    |
| `weight`        | safe integer from 1 through 100     | Deterministic penalty weight; it does not override hard semantics. |

Valid combinations and effects:

| Requirement | Connection | Effect                                                                                                                    |
| ----------- | ---------- | ------------------------------------------------------------------------------------------------------------------------- |
| `required`  | `door`     | Same-floor rooms must share a non-zero divider, and one door is placed in it.                                             |
| `required`  | `near`     | Enforces the deterministic near-distance threshold, applies the weighted centroid-distance score, and creates no opening. |
| `preferred` | `door`     | Missing direct adjacency adds a penalty; adjacency creates one divider door.                                              |
| `preferred` | `near`     | Integer centroid distance adds a weighted soft penalty and creates no opening.                                            |
| `avoid`     | `none`     | The pair must not share a direct divider.                                                                                 |

`avoid` with `door` or `near` is invalid. Door relationships cannot cross floors. Rooms opposite
across the corridor are not directly adjacent. Relationships without this directive are preserved
during emission but do not constrain planner v0.1. An unordered room pair may have at most one valid
architecture adjacency directive across all `door`, `near`, and `none` connections, so duplicate or
conflicting instructions are rejected deterministically.

For a solved corridor length of `L` grid cells, the v0.1 near threshold is
`T = max(1, floor(L / 2))` grid cells. To avoid fractional centroids, evaluation uses doubled-cell
Manhattan distance:
`D2 = abs((2x1 + w1) - (2x2 + w2)) + abs((2z1 + d1) - (2z2 + d2)) + 2 * abs(level1 - level2) * L`. A
near relationship is satisfied exactly when `D2 <= 2T`. `required + near` rejects a candidate beyond
that threshold with `architecture.required_adjacency_unsatisfied`; both required and preferred near
relationships still contribute `D2 * weight` to `score.nearDistance`.

## Supported source WorldSpec profile

A source must pass `@worldwright/worldspec` validation before profile extraction. The selected
building profile requires:

- one `world` root;
- optional `region` and `parcel` ancestors;
- exactly one `structure` carrying the building directive;
- one through three direct floor children;
- at least two direct room children per floor;
- one entrance room on level zero;
- one direct stair route for a multi-floor building; and
- no unsupported descendants within the structure.

Input entities must not contain `worldwright.roblox`. No source identifier may begin `archgen-`. A
lock affecting the planned building fails with `architecture.lock_unsupported`; it is never deleted.
Planner v0.1 supports only a `reachability` constraint with non-empty room-only `subjectIds` and
`targetIds` and an empty `parameters` object; evaluation proves each subject can reach each target
through the explicit undirected circulation graph. Unsupported error-severity constraints affecting
the building fail. Unsupported warnings are preserved and reported as unevaluated. Arbitrary
constraints are never claimed as satisfied.

## Coordinate and grid model

Horizontal combinatorial state uses safe integer cells. Dimensions that are required to align are
divided by `gridSize` and must yield safe integers exactly. Solver comparisons and allocation happen
in cells; the selected plan reports stud coordinates.

A local rectangle has minimum corner `x`, `z` and positive `width`, `depth`. The footprint is
centered at local zero. The corridor spans the full interior length along its selected axis and is
centered in the perpendicular dimension. After corridor-wall volume is reserved, remaining cells
form two room bands. An indivisible extra cell is assigned by the documented canonical candidate
rule.

Room, corridor, and stair-hall rectangles are clear space. Exterior walls occupy the difference
between the outer footprint and interior envelope. Divider thickness is counted between consecutive
rooms. World emission applies exact yaw 0/90/180/270 coordinate mappings; it never uses trigonometry
or composes parent transforms.

## Topology and solver

`double_loaded_spine` contains one straight corridor, two exterior-facing room bands, one corridor
door per room, and an optional rear stair hall containing a corridor-side access lane and aligned
stair core.

Global candidates cover any requested `auto` corridor axis and stair side plus the canonical
unequal-band split alternative. Per floor, room expansion order is required-degree descending,
minimum-area descending, then source ID. A beam state may put the next room on either side and at
either sequence end. The entrance is forced to an entrance-facing end.

The beam width is fixed at **256**. States that cannot fit remaining minimum lengths are pruned.
Canonical partial-state ordering uses the absolute difference between the two bands' assigned
preferred-minus-minimum length slack, then canonical signature. This capacity-balance heuristic is
deterministic but is not an admissible lower bound on final primary score. The seed tie key is
excluded from partial pruning and is considered only for complete plans whose non-seed score
components are exactly equal. The search is bounded, iterative, independent of object iteration
order, and contains no `Math.random` or clock input.

Permissive room maximum-area and maximum-aspect values are clamped to the supported 4,096-cell band
length when constructing allocation limits. Widening a maximum beyond candidate capacity therefore
cannot make an otherwise feasible program fail.

After a side sequence is chosen, room lengths begin at minimum feasible cells. Remaining capacity is
distributed toward preferred area using deterministic largest deficit, then toward maximum area.
Source ID is the final allocation tie break. Divider and stair-wall thicknesses are explicit. A
candidate fails unless the authored rooms exactly fill available room-band capacity without an
invented filler or unexplained void.

## Hard rules

A successful plan proves all of the following through semantic evaluation:

- room containment, non-overlap, area, span, and aspect limits;
- corridor and exterior contact plus one valid corridor door per room;
- entrance facade contact and one exterior entrance door;
- shared divider geometry for required-door relationships;
- the exact deterministic centroid-distance threshold for required-near relationships;
- no shared divider for avoided pairs;
- valid, non-overlapping, vertically bounded openings and all minimum windows;
- continuous corridor, aligned stair core, fitting stair runs, and connected floors;
- explicit circulation reachability from outside to every room;
- globally valid and unique IDs;
- deterministic ordering and exact metrics; and
- output within the declared instance budget when present.

## Score

Hard-feasible candidates receive an integer penalty breakdown:

| Field                | Meaning                                                                 |
| -------------------- | ----------------------------------------------------------------------- |
| `total`              | Incrementally saturated sum of all non-seed primary components.         |
| `areaDeviation`      | Absolute deviation from preferred clear area.                           |
| `aspectRatio`        | Penalty for less desirable but allowed proportions.                     |
| `preferredAdjacency` | Penalty for unsatisfied preferred-door relationships.                   |
| `preferredWindows`   | Sum of preferred minus fitting exterior windows, floored at zero.       |
| `nearDistance`       | Weighted integer centroid-distance penalty for near relationships.      |
| `zoneOrdering`       | Public/private/service placement penalty relative to entrance and rear. |
| `seedTieBreak`       | Deterministic SHA-256 tie value; not part of the primary quality total. |

`preferredWindows` is included directly in `total`; corridor-axis preference is included in
`zoneOrdering`. A different project seed may select another candidate only when all non-seed score
components are equal. Every score component and the non-seed total saturate at the maximum safe
integer, `9007199254740991`. The total uses incremental saturating addition, so permissive soft
targets cannot create unsafe intermediate arithmetic or disagree with plan validation.

## Architecture Plan contract

The top-level object contains these required fields:

| Field              | Meaning                                                                           |
| ------------------ | --------------------------------------------------------------------------------- |
| `schemaVersion`    | Architecture Plan schema version `0.1.0`.                                         |
| `plannerVersion`   | Planner implementation version `0.1.0`.                                           |
| `source`           | Exact canonical source identity and hash.                                         |
| `building`         | Selected normalized building, coordinate, dimension, material, and color profile. |
| `floors`           | Canonically ordered per-floor topology records.                                   |
| `spaces`           | Strict room, corridor, and stair-hall placement union.                            |
| `walls`            | Canonical logical wall segments.                                                  |
| `openings`         | Door and window intervals resolved to walls and connection endpoints.             |
| `stairRuns`        | One straight-run record per adjacent floor pair.                                  |
| `circulationEdges` | Explicit undirected navigability edges.                                           |
| `metrics`          | Exact deterministic hard-rule and output-count measurements.                      |
| `score`            | Integer primary penalty breakdown and seed tie value.                             |

The untrusted plan boundary rejects oversized collections before semantic or pairwise geometry work.
Top-level limits are 3 floors, 102 spaces, 309 walls, 6,756 openings, 2 stair runs, and 614
circulation edges. Nested reference limits are 34 spaces, 103 walls, 2,594 openings, and 2 stair
runs per floor; 3 exterior walls per room; and 65 openings per wall. These bounds follow from the
supported three-floor, 32-room, 64-window, and 512-relationship profile and are also encoded as
`maxItems` in the generated Architecture Plan schema.

### Source record

`source` records the WorldSpec schema version, project ID, lowercase SHA-256 of canonical source
WorldSpec, and selected building entity ID. The source hash is recomputed before emission.

### Building record

`building` records the selected topology, outer footprint, interior envelope, local origin
convention `footprint_center`, world origin, yaw, grid size, corridor axis, entrance end, wall and
slab dimensions, opening defaults, and closed material, color, and window-transparency profile.

### Floor records

Each floor records its source floor ID, level, world finished-floor elevation, clear height,
footprint, corridor clear rectangle, optional stair-core rectangle, and ordered IDs for its spaces,
walls, openings, and stair runs.

### Space union

- A `room` uses its source room ID and records floor ID, local clear rectangle, zone, entrance flag,
  source provenance classification, corridor-door opening ID, exterior wall IDs, actual clear area,
  and aspect ratio.
- A generated `corridor` records its deterministic ID, floor ID, and local clear rectangle.
- A generated `stair_hall` records its deterministic ID, floor ID, rectangle, and source stair route
  ID.

### Logical walls

Every wall records ID, floor ID, `kind` (`exterior`, `corridor`, `divider`, or `stair`), axis (`x`
or `z`), constant coordinate, ordered `start` and `end`, thickness, height, applicable adjacent
space IDs, exterior marker, and ordered opening IDs. Wall direction is canonical and duplicate or
overlapping logical segments are invalid.

### Openings

Every opening records ID, floor ID, wall ID, `type` (`door` or `window`), offset from canonical wall
start, width, bottom height, height, source room or relationship ID, and its two connection endpoint
IDs. An interval stays within the wall with end clearance, does not overlap another interval, and
remains vertically within wall height.

### Stair runs

Each run records ID, source stair route ID, from-floor and to-floor IDs, core rectangle, direction,
step count, riser height, tread depth, clear width, and landing geometry. Step count is
`ceil(rise / maximumRiserHeight)`; riser and tread values must remain finite and within their hard
limits, and `stepCount` may not exceed 256. Landing depth is
`max(minimumTreadDepth, min(coreLength / 4, 4))`; two such depths are reserved before dividing the
remaining run length among steps. The four-stud cap keeps a radius-two traversal envelope clear at
the center of a retained landing when the authored core is feasible.

### Circulation edges

Each edge records deterministic ID, source opening or stair-run ID, from-node ID, to-node ID, and
traversal type `door`, `open`, or `stair`. The graph is undirected even though each edge has one
canonical endpoint order.

### Metrics

Metrics include floor and room counts; gross outer, clear-room, corridor, and stair areas;
clear-area efficiency; required, preferred, and avoided adjacency totals and satisfied counts;
maximum room aspect ratio; door, window, and stair-run counts; `allRoomsReachable`; and estimated
derived WorldSpec entity and primitive counts. Values are recomputed during evaluation rather than
trusted from the document.

## Walls, openings, and physical panels

Logical walls are generated before physical parts. They are split at space boundaries,
canonicalized, deduplicated, and checked for conflicting overlap. Openings are sorted by offset and
subtracted into full-height intervals plus lower and upper panels around each opening. Every emitted
panel has positive finite dimensions and exact wall thickness. Panel coverage equals wall area minus
opening area; panels do not overlap.

Generated wall-panel and window-glass entity IDs remain the stable semantic identity. Their
Roblox-facing display names use concise one-based ordinals in canonical emission order
(`Wall Panel 1`, `Window Glass 1`, and so on), keeping planner-generated `Instance.Name` values
within Roblox's 100-character limit without encoding identity in a display field. Authored entity
names remain unchanged semantic input and are validated separately by a live adapter before Studio
mutation.

Every room has one centered-or-deterministically-shifted corridor door. The entrance has one facade
door. Satisfied door relationships create one divider opening. Every room has an exterior wall and
receives minimum windows before preferred windows. Windows are evenly distributed and respect end
clearance, vertical bounds, and the entrance door.

Window openings emit non-collidable Glass panes with `canCollide: false`, `canTouch: false`, and
`canQuery: true`. The fixed v0.1 pane rule is `castShadow: false`; door openings remain empty. No
moving door or script is emitted.

## Slabs and stairs

The lowest slab covers the complete outer footprint and extends downward from finished floor. Upper
slabs use pure rectangle subtraction to remove the complete aligned stair core and form
non-overlapping positive panels. The lowest starting landing is a separate primitive immediately
above the complete base slab. Each upper arrival landing is a separate primitive inside the opening,
flush with the surrounding slab surface, and is not duplicated as a slab panel. A shared
arrival/departure rectangle in a three-floor stair is emitted once per floor and rectangle.

There is one run per adjacent floor pair. Step count is positive, riser height does not exceed its
maximum, tread depth meets its minimum, no run exceeds 256 steps, and all steps fit the core and
landing. Generated steps and landings are anchored Block Parts. They are blockout geometry, not
certified stairs.

## Circulation evaluation

The evaluator constructs the graph from explicit entrance, corridor, room-divider, stair-hall, and
stair connections. Iterative search begins at the exterior entrance. Every room, corridor, stair
hall, and floor must be visited. Rectangle contact without an opening creates no edge.

## Generated IDs

`archgen-` is reserved for planner output. Source IDs using it fail. Generated IDs:

- satisfy the WorldSpec identifier pattern and 128-character maximum;
- derive from stable semantic IDs and roles, never array position alone;
- are ordered by Unicode code point;
- use a readable prefix plus lowercase SHA-256 fragment when truncation is necessary;
- resolve collisions deterministically; and
- never use a random UUID, system time, machine path, or locale-sensitive operation.

Source room placements keep their source IDs. The reservation applies to generated corridors, stair
halls, logical groups, openings, wall and slab panels, panes, steps, and landings.

## Normalization and hashing

Architecture Plan normalization returns a deep-independent value with contract-defined array order,
recursively code-point-sorted JSON object keys, and normalized negative zero. Serialization uses two
spaces, one final line feed, and platform-independent line endings.

The source hash is SHA-256 over canonical `stringifyWorldSpec(validatedSource)`. The plan hash is
SHA-256 over canonical Architecture Plan JSON. Both are lowercase hexadecimal. A hash is an
integrity link, not creator authentication, authorization, or a digital signature.

## WorldSpec emission

`emitArchitectureWorldSpec` validates source and plan, recomputes the source hash, rejects stale
plans, performs complete plan evaluation, and returns a deep-independent derived WorldSpec.

The emitter preserves source project, intent, references, style, budgets, semantic identities,
names, provenance, tags, relationships, supported constraints, and locks. It adds fixed container
directives to source world/region/parcel/structure/floor/room/route entities. It adds semantic
entities for corridors, stair halls, openings, wall groups, slab groups, and stair runs, plus one
primitive entity for each wall panel, slab panel, pane, step, and landing.

Every primitive has a world-space transform, positive bounds, architecture linkage, strict
allowlisted `worldwright.roblox` data, and invented provenance describing deterministic blockout
generation. The structure receives compact source and plan hash metadata, not the complete plan.

The derived WorldSpec is validated and compiled through the public Roblox compiler. Emission returns
success only if compilation contains no error diagnostics. An exact output entity count is checked
against `budgets.limits.instances` before success. Independently, an exact preflight rejects more
than 16,384 generated entities or 12,288 generated primitives before generated entities are
expanded.

## Public result model

Public operations use discriminated data results:

- directive and plan validators return either `{ valid: true, value, diagnostics }` or
  `{ valid: false, diagnostics }`;
- planning success contains a normalized Architecture Plan and warning diagnostics;
- evaluation success contains a normalized plan and recomputed measurements;
- emission success contains the derived WorldSpec, compiled Manifest, plan hash, and diagnostics;
  and
- combined planning and emission success contains both artifacts and compilation evidence.

Expected invalid source, directive, geometry, stale-plan, budget, and compilation outcomes do not
require exception handling. All successful returned contracts are independent of caller-owned input.

| `architecture.generated_id_collision`         | Deterministic ID derivation could not produce a unique generated ID.       |
| `architecture.emission_invalid`               | Derived WorldSpec validation or compiler verification failed.              |
| `architecture.preference_unsatisfied`         | A valid plan could not satisfy an important soft preference; warning only. |

Wrapped WorldSpec and compiler failures retain safe stable context without exposing Ajv text, stack
traces, or hostile file contents.

## CLI

Run the source CLI from the repository root:

```sh
pnpm architecture-planner plan <source-worldspec>
pnpm architecture-planner plan <source-worldspec> --output <plan-file>
pnpm architecture-planner plan <source-worldspec> --json

pnpm architecture-planner emit <source-worldspec> --plan <plan-file>
pnpm architecture-planner emit <source-worldspec> --plan <plan-file> --output <derived-worldspec-file>
pnpm architecture-planner emit <source-worldspec> --plan <plan-file> --json

pnpm architecture-planner build <source-worldspec>
pnpm architecture-planner build <source-worldspec> --plan-output <plan-file> --worldspec-output <derived-worldspec-file>
pnpm architecture-planner build <source-worldspec> --json
```

`build` is deterministic convenience for plan plus emit. Artifact output is canonical and occurs
only with an explicit output path. Source and plan inputs cannot be overwritten by an output path.
JSON mode includes warnings and diagnostics in stable machine-readable form.

| Exit | Meaning                                                                                      |
| ---- | -------------------------------------------------------------------------------------------- |
| `0`  | Success, including success with warnings.                                                    |
| `1`  | Valid command, but source, plan, architecture, budget, emission, or compilation was invalid. |
| `2`  | Usage, unsupported option or command, or file I/O error.                                     |

Expected failures do not print stack traces. The CLI performs no network access or shell composition
and offers no command that applies anything to Roblox Studio.

## Security and integrity

The planner accepts only ordinary JSON data and allowlisted compiler directives. It introduces no
functions, scripts, dynamic evaluation, asset IDs, generated content URLs, arbitrary Roblox class,
arbitrary property setter, network call, AI provider, database, authentication, telemetry, or
deployment integration.

Search is bounded independently of user preferences. Arithmetic is finite and checked. Planning and
emission do not mutate caller input. Emission rejects stale source hashes, lock conflicts,
unsupported hard constraints, reserved-ID conflicts, generated collisions, and instance-budget
overflow. Canonical artifacts contain no timestamp, random UUID, credential, private fixture
content, absolute machine path, or platform-specific line ending.

## Performance

The exact v0.1 bounds are:

| Work dimension                         | Limit       |
| -------------------------------------- | ----------- |
| Floors                                 | 3           |
| Rooms per floor                        | 32          |
| Architecture relationship directives   | 512         |
| Architecture Plan spaces               | 102         |
| Architecture Plan logical walls        | 309         |
| Architecture Plan openings             | 6,756       |
| Architecture Plan stair runs           | 2           |
| Architecture Plan circulation edges    | 614         |
| Along-band grid length                 | 4,096 cells |
| Beam states retained                   | 256         |
| Complete candidates retained per floor | 64          |
| Global floor combinations evaluated    | 512         |
| Windows requested per room             | 64          |
| Steps expanded per stair run           | 256         |
| Generated entities                     | 16,384      |
| Generated primitives                   | 12,288      |

Each beam expansion has deterministic polynomial validation and sorting work. Geometry expansion is
linear in selected walls, openings, slab panels, and stair steps after layout selection, and exact
generated counts are checked before that entity expansion.

The planner makes no wall-clock guarantee and does not use flaky timing as correctness evidence. A
bounded beam can sacrifice completeness: failure means no feasible candidate survived the defined
search, not a mathematical proof that no layout exists under every possible ordering.

## Limitations

Version `0.1.0` produces a rectangular double-loaded-spine blockout only. It has no basement, split
level, multiple core, wing, courtyard, curved, terrain-aware, roof, facade-detail, furniture,
lighting, gameplay, accessibility-certification, structural, fire-egress, acoustic, or building-code
model. Stairs are blockout approximations. Windows are simple Glass Parts. Doors are empty openings.

It does not infer a plan from names or images. It does not claim that hidden geometry is observed.
It does not produce finished visual art or guarantee Roblox avatar traversal. It does not connect to
Studio, Forge, Studio MCP, ChangeHistoryService, or any external service.

## Versioning

Unsupported schema versions are rejected. Patch releases may correct code or documentation without
changing accepted shape or meaning. Backward-compatible fields or variants require a new minor
contract and generated schema. Removal, renaming, narrowing, or reinterpretation requires a major
version.

Coordinate semantics, grid conversion, topology, beam width, candidate ordering, allocation,
scoring, seed behavior, generated-ID derivation, wall orientation, opening semantics, circulation
rules, canonical array order, and hash inputs are observable behavior. Changes to them require an
intentional version decision and synchronized schemas, fixtures, tests, and documentation.

Architecture Planner `0.1.0` consumes WorldSpec `0.1.0` and emits compiler-compatible WorldSpec
`0.1.0` without changing the WorldSpec or Roblox compiler wire contracts.
